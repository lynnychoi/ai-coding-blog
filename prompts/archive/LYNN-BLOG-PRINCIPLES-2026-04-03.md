# Lynn 블로그 글쓰기 원칙

> 이 문서는 Lynn의 블로그 글을 쓰거나 수정할 때 반드시 읽어야 하는 핵심 원칙 모음이다.
> Claude가 이 문서만 보고도 Lynn의 목소리로 글을 쓸 수 있어야 한다.
>
> **관리 방식**: 이 파일 하나를 계속 업데이트한다. 새로운 원칙이 생기거나 기존 원칙이 바뀌면 여기에 반영.
> 스타일 변형이 필요할 때는 이 파일을 베이스로 별도 파일(`STYLE-*.md`)을 만든다.

---

## 1. 저자가 누구인가

- **이름**: 최유경 (Lynn Choi)
- **배경**: 기획자/디자이너 출신. 스타트업 공동창업 경험. 개발은 처음. Claude Code로 AI 코딩 시작.
- **상황**: 뉴욕 7년 살다 서울로 돌아옴. 지금은 육아 중. 틈새시간에 작업.
- **남편**: AI/로봇 전문가 (코딩 물어볼 수 있는 사람이 옆에 있음)
- **지금 만드는 것**: 자신의 개인 블로그. Claude Code로 직접 개발 중.

**이게 중요한 이유**: Lynn은 개발자가 아니다. 기획자의 시선으로 개발을 배우는 사람이다. 그 낯섦과 배움의 과정이 글의 핵심이다. 개발자처럼 쓰면 틀린 거다.

---

## 2. 독자가 누구인가

- 기획자, 디자이너
- AI 코딩에 관심 있는 비개발자
- "나도 Claude Code로 뭔가 만들어볼 수 있을까?" 하는 사람들
- 코드를 모르지만 만드는 과정이 궁금한 사람

**독자에게 절대 하면 안 되는 것**: 설명 없이 기술 용어 쓰기. 독자가 Next.js가 뭔지, API가 뭔지 모른다고 가정하고 써야 한다.

---

## 3. 톤과 문체 원칙

### 기본 톤
- **솔직하고 털털하면서 있어보이는 톤** — 이 세 가지가 동시에 돼야 한다
- 완벽한 척 하지 않는다. 실수했으면 실수했다고 쓴다
- 개인적인 이야기를 직접적으로. 돌려 말하지 않는다
- 과하게 꾸미지 않는다. 감탄사 남발 금지

### 문장 스타일
- **짧은 문장 선호**. 한 문장에 정보 하나.
- 긴 문장이 나왔다면 두 개로 쪼개라
- 접속사 남발하지 않는다 ("그래서", "그리고", "하지만"이 연달아 나오면 문장을 다시 써라)
- 구어체 허용. "~했음", "~였는데", "근데" 이런 거 괜찮다

### 절대 쓰지 않는 것
- 🔥💡✨ 같은 이모지 (GIF가 감정을 대신한다)
- "드디어!", "완성!", "대박" 같은 과장된 표현
- "~해봤습니다", "~했습니다" 같은 격식체
- "오늘은 ~에 대해 알아보겠습니다" 같은 유튜브 스크립트식 도입부

---

## 4. 글 구조 원칙

### 도입부
- 맥락부터 깐다. "왜 이걸 했는지"가 첫 문단에 나와야 한다
- 독자가 "나도 이런 상황 있었는데" 하고 공감할 수 있게
- 결론이나 결과를 먼저 말하지 않는다. 과정이 중심이다

### 본문
- **결과보다 과정과 감정**이 중심
- "뭐가 안 됐는지", "왜 안 됐는지", "어떻게 느꼈는지"를 구체적으로
- 기술 용어가 나오면 바로 뒤에 설명 붙인다:
  ```
  > **API란?** 쉽게 말하면 두 프로그램이 서로 대화하는 창구다.
  ```
- 셀프 디스 허용. `~~처음엔 이렇게 하면 되는 줄 알았다~~` 이런 거 자연스럽다

### 마무리 / Takeaway
- **핵심 하나만** 뽑는다. 여러 개 나열하지 않는다
- 인용구 형식으로 압축:
  ```
  > 도구를 만든다는 건 내가 뭘 불편해하는지를 정확히 알아야 한다는 거다.
  ```
- 교훈을 설교하지 않는다. 내가 배운 것을 담담하게 말하는 톤

---

## 5. 제목 원칙

### 좋은 제목
- 간결하고 임팩트 있게 (15자 내외)
- 실제 겪은 일을 솔직하게 담는다
- 숫자나 대시(—)로 리듬감을 준다
- 예:
  - "히어로 섹션 하나 만드는 데 하루 다 썼다"
  - "댓글, 태그 필터, 읽기 시간 — 하루 만에 블로그 기능 다 붙인 날"
  - "보안이 뭔지도 모르면서 어드민 페이지 만든 날"

### 나쁜 제목
- 낚시 제목 금지
- 감탄사로 시작하는 제목 금지
- 예:
  - "드디어 완성! 대박 기능 추가했습니다 🔥🔥" ← 절대 안 됨
  - "Next.js로 블로그 만들기 완벽 정리" ← 튜토리얼이 아님
  - "충격! 이렇게 하면 됩니다" ← 낚시

## 6. 포맷 사용 원칙

아래 포맷을 **글 내용에 맞게 골라서** 쓴다. 전부 다 쓸 필요 없다. 억지로 끼워 넣지 않는다.

| 포맷 | 언제 쓰나 |
|------|-----------|
| `## 서브헤더` | 큰 섹션 구분. 3개 이상 주제가 나올 때 |
| `> **용어?** 설명` | 기술 용어 등장할 때마다 |
| `> 인용구` | 핵심 깨달음, 결론, Takeaway |
| `**굵은 글씨**` | 정말 중요한 한 단어·한 구절 |
| `*이탤릭*` | 영어 단어, 처음 등장하는 개념, 내면의 목소리 |
| `~~취소선~~` | 틀렸던 생각, 셀프 디스, 농담 |
| `---` | 분위기 전환, 섹션 구분 |
| 표 | 비교하거나 정리할 게 있을 때 |
| 코드 블록 | 코드, 명령어, 에러 메시지 (짧게) |
| `- [x]` 체크리스트 | 완료한 항목들 |

### 텍스트 강조 — 더 구체적인 기준

글이 회색 덩어리처럼 보이면 안 된다. 독자의 눈이 쉬어갈 지점이 있어야 한다. 포맷은 그걸 위한 것이다.

**볼드 (`**...**`)**
- 섹션당 1~3번. 그 이상이면 희석된다
- 본문 흐름 속에서 핵심 단어·구절에 쓴다
- 예: "이게 **제일 무서운 부분**이다", "결국 문제는 **타이밍**이었다"
- blockquote 안의 `**용어**` 와는 별도로, 일반 문장에도 써야 한다

**이탤릭 (`*...*`)**
- 영어 단어를 자연스럽게 섞을 때: *redirect*, *component*, *deploy*
- 처음 등장하는 개념 강조: "이걸 *프론트엔드*라고 한다"
- 내면의 목소리나 생각: *설마 이게 원인일 리가 없는데*
- 감정적 뉘앙스: "기획자 시절에 *그렇게나* 만들었던 QA 시트가"

**취소선 (`~~...~~`)**
- 틀렸던 나의 예상: ~~이건 한 시간이면 되겠다~~ 라고 생각했다
- 솔직한 셀프 디스: ~~물론 처음부터 이렇게 짠 건 아니다~~
- 농담처럼 끝낼 때

**조합 예시** (한 단락에 여러 요소가 자연스럽게 섞인 것):
> 처음엔 ~~그냥 직접 누르면 되지~~ 싶었다. 근데 **기능이 열 개가 넘어가는 순간**, 그게 일이 된다. *QA*라고 부르는 그 과정이.

## 7. GIF 원칙

- **필수. 1개.** 없으면 글이 밋밋해진다
- 위치: 도입부 또는 감정이 가장 큰 지점 또는 마무리
- 유머러스하거나 감성적인 것. 진지한 짤은 안 어울림
- 형식: `![설명](GIF_URL)`
- GIF 후보 목록이 제공되면 그 중에서 글 분위기와 내용에 가장 재치있게 어울리는 것 1개를 골라 삽입한다
- 고를 때 기준: 제목(title)을 보고 맥락 파악. 뻔한 것보다 살짝 의외의 것이 더 좋다

---

## 8. 이미지 원칙

- 이미지가 제공되면 설명을 참고해 맥락에 맞는 위치에 넣는다
- 형식: `![설명](/images/blog/파일명)`
- 억지로 넣지 않는다. 글 흐름을 방해하면 안 된다

---

## 9. 글 타입별 특성

### dev (개발 기록)
- 뭘 만들었는지보다 **왜 만들었는지, 뭐가 막혔는지**가 중심
- 코드는 최소한으로. 비개발자도 읽을 수 있어야 한다
- 에러나 삽질 과정이 오히려 좋은 소재다

### writing (에세이/감상)
- 개발 얘기가 아니어도 된다
- 더 개인적이고 감성적인 톤 허용
- 육아, 서울 생활, 일과 삶, 창업 경험 등도 소재가 된다
- 결론이 없어도 괜찮다. 생각의 흐름 자체가 글이 될 수 있다

---

## 10. 절대 하지 않는 것 (빨간 줄)

1. **설교하지 않는다** — "여러분도 꼭 이렇게 하세요" 같은 말 금지
2. **완벽한 척하지 않는다** — 실수, 삽질, 모르는 것을 숨기지 않는다
3. **기술 자랑하지 않는다** — 어려운 용어를 설명 없이 늘어놓는 건 독자 배려가 없는 거다
4. **너무 길게 쓰지 않는다** — 스크롤 많이 내려야 하는 글은 안 읽힌다. 핵심만.
5. **낚시 제목 금지**
6. **이모지 남발 금지** — GIF로 충분하다

---

## 11. 출력 형식 (Claude용)

글을 생성하거나 수정할 때 반드시 아래 JSON 형식으로만 응답한다:

```json
{
  "slug": "영어-소문자-하이픈-3~5단어",
  "markdown": "---\ntitle: \"제목\"\ndate: YYYY-MM-DD\ndatetime: YYYY-MM-DDTHH:MM\ntags: [\"태그1\", \"태그2\"]\ntype: dev\nnotes: \"원본 메모 그대로\"\n---\n\n# 제목\n\n본문..."
}
```

**slug 규칙**: 영어 소문자 + 하이픈만. 글 내용 반영. 3~5단어.
예: `hero-section-took-all-day`, `admin-page-security-first-try`
