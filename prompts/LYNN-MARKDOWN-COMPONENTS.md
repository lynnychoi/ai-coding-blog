# Lynn 블로그 마크다운 컴포넌트 가이드

> 이 블로그의 마크다운은 일반 마크다운과 다르게 렌더링되는 전용 컴포넌트가 있다.
> 글을 쓸 때 이 파일을 참고해서 적절한 컴포넌트를 골라 써야 한다.
> 컴포넌트를 안 쓰면 글이 회색 덩어리가 된다. 과하게 쓰면 글이 산만해진다.
> **문법이 정확해야 렌더링된다.** 특히 def-box와 takeaway-box는 형식이 틀리면 그냥 일반 blockquote로 보인다.

---

## 컴포넌트 목록 한눈에 보기

| 컴포넌트 | 문법 | dev | writing |
|---------|------|:---:|:-------:|
| h1 제목 | `# 제목` | ✅ | ✅ |
| h2 섹션 | `## 소제목` | ✅ | ✅ |
| h3 하위 섹션 | `### 소소제목` | ✅ 남용 금지 | ✅ 남용 금지 |
| blockquote | `> 문장` | ✅ | ✅ |
| **def-box** | `> 단어란? 설명` | ✅ | 드물게 |
| **takeaway-box** | `> !! 내용` | ✅ | ✅ |
| 순서 없는 목록 | `- 항목` | ✅ | 드물게 |
| 순서 있는 목록 | `1. 항목` | ✅ | 드물게 |
| 굵게 | `**텍스트**` | ✅ | ✅ |
| 기울임 | `*텍스트*` | ✅ | ✅ |
| 취소선 | `~~텍스트~~` | ✅ | ✅ |
| 밑줄 | `<u>텍스트</u>` | ✅ | ✅ |
| 인라인 코드 | `` `코드` `` | ✅ | 드물게 |
| 코드 블록 | ` ```언어 ``` ` | ✅ | ❌ |
| 표 | `\| 헤더 \|` | ✅ | ❌ |
| 이미지/GIF | `![캡션](URL)` | ✅ | ✅ |
| 구분선 | `---` | ✅ | ✅ |

## 각 컴포넌트 상세 가이드

---

### 1. 헤딩 (`#`, `##`, `###`)

**렌더링**: h2는 위에 민트(dev) 또는 라벤더(writing) 가로선이 그려진다. h3는 선 없음.

**h1 — 글 제목**
```markdown
# 히어로 섹션 하나 만드는 데 하루 다 썼다
```
- 글 전체에서 딱 한 번. 파일 맨 위.
- 제목 바로 아래에는 산문 없이 바로 `##` 섹션이 와야 한다.

**h2 — 섹션 구분** ← 가장 중요
```markdown
## 이게 맞나 싶었는데
```
- 주제가 바뀔 때마다 반드시 쓴다. 최소 2~3개는 있어야 한다.
- `##` 없이 글이 이어지면 독자 입장에서 회색 덩어리로 보인다.
- 섹션 제목도 글 제목처럼 만든다. "과정 설명" 같은 밍밍한 제목 금지.

**h3 — 하위 구분**
```markdown
### 삽질 1: 블롭이 한쪽에서만 맴돌았다
```
- `h2` 안에서 세부 항목이 여러 개일 때만. 남용 금지.
- 삽질이 여러 개일 때 번호 붙여서 쓰는 패턴에 잘 맞음.

---

### 2. blockquote (`>`)

**렌더링**: 왼쪽 세로선이 생기는 인용 스타일.

```markdown
> 만드는 것보다 만든 걸 안 깨뜨리는 게 더 어렵다.
```

**언제**: 글의 핵심 깨달음, 인용, 감정이 응축된 한 문장.
**왜**: 본문 흐름에서 이 문장만 속도를 늦추고 싶을 때. 독자 눈이 여기서 잠깐 멈춘다.
**주의**: 글 하나에 1~2개. 많아지면 강조 효과가 없어진다.

❌ 쓰면 안 되는 경우:
```markdown
> 오늘은 Playwright를 설치했다.   ← 평범한 정보는 blockquote 아님
```

---

### 3. def-box (`> 단어란?` / `> 단어이란?`)

**렌더링**: 특별한 카드 형태로 자동 변환됨.
**⚠️ 반드시 `란?` 또는 `이란?`으로 시작해야 작동한다. 다른 형식은 일반 blockquote로 보임.**

```markdown
> Playwright란? 브라우저를 코드로 자동 조작하는 도구다. 내가 직접 클릭하는 걸 로봇이 대신 해준다.
> 리다이렉트란? 어떤 페이지에서 다른 페이지로 자동으로 넘겨주는 것.
> API이란? 두 프로그램이 서로 대화하는 창구다.
```

**언제**: 기술 용어가 처음 등장할 때. 독자가 모를 수 있는 단어 직후에 바로 붙인다.
**왜**: 비개발자 독자를 위한 것. 글 흐름을 끊지 않고 설명을 넣을 수 있다.

**핵심 규칙:**
- 글 하나에 **최대 3개**. 그 이상이면 교과서처럼 읽힌다
- 설명은 1~2문장으로 짧게. 비유를 쓰면 좋다
- 같은 용어를 두 번 설명하지 않는다
- 이 용어를 몰라도 글 흐름을 따라갈 수 있다면 → 그냥 넘어간다
- writing 글에는 기술 용어가 거의 없으니 거의 쓸 일 없음

❌ 이렇게 쓰면 카드로 변환 안 됨:
```markdown
> **API란?** 설명...        ← ** 붙이면 안 됨
> API는 무엇인가? 설명...   ← "란?" 형식 아님
> API란 설명...             ← "란?" 뒤에 ? 없으면 안 됨
```

✅ 이렇게 써야 함:
```markdown
> API란? 두 프로그램이 서로 대화하는 창구다.
> Next.js이란? React 기반의 웹 프레임워크다.
```

---

### 4. takeaway-box (`> !!`)

**렌더링**: 배경색이 들어간 강조 카드로 자동 변환됨.
**⚠️ `!!`로 시작하는 blockquote만 변환된다. `> !!` 이후 바로 내용이 와야 함.**

```markdown
> !! 만들었다고 쓰이는 게 아니다. 코드가 읽고 있는지 확인해야 한다.
```

**언제**: 글의 핵심 한 줄. 이 글에서 독자가 가져가야 할 것이 있다면 여기에 넣는다.
**왜**: 가장 눈에 띄는 컴포넌트. 글에서 제일 기억에 남아야 할 문장에만 쓴다.

**핵심 규칙:**
- 글 하나에 **딱 1개**. 절대 2개 이상 쓰지 않는다
- 위치는 자유. 마지막에 올 필요 없다. 글의 핵심이 중간에 나와도 된다
- 교훈을 설교하듯 쓰지 않는다. 내가 배운 것을 담담하게
- writing 글에서도 쓸 수 있다. 단 결론이 있는 글에서만
- 문장이 길면 안 된다. 한 줄, 혹은 두 문장 이내

❌ 잘못된 사용:
```markdown
> !! 오늘은 Playwright를 배웠다.      ← 평범한 사실 정리 아님
> !! 이렇게 하면 됩니다!              ← 설교 톤 금지
```
```markdown
> !! 첫 번째 배운 것
> !! 두 번째 배운 것                  ← 2개 이상 절대 금지
```

✅ 잘된 사용:
```markdown
> !! 파일을 만들어놓는 것과 코드에서 실제로 읽는 것은 다른 일이다.
> !! 완성된 걸 배포하는 것보다, 내일도 열 수 있는 게 더 중요하다.
```

### 5. 인라인 텍스트 강조

**굵게 (`**텍스트**`)**
```markdown
**기획자가 코드를 짜면** 생기는 일
새벽 **11시 32분**, 아이가 잠들었다.
```
- 핵심 용어, 강조 포인트. 섹션당 1~3개 이내
- 그 이상이면 강조 효과가 희석된다

**기울임 (`*텍스트*`)**
```markdown
*"설마 이게 원인일 리가 없는데"* 싶었는데 맞았다.
*redirect*, *component* 같은 영어 단어
기획자 시절에 *그렇게나* 만들었던 QA 시트가
```
- 속마음, 혼잣말, 내면의 목소리
- 영어 단어를 자연스럽게 섞을 때
- 뉘앙스 강조

**취소선 (`~~텍스트~~`)**
```markdown
~~한 시간이면 되겠다~~ 라고 생각했다.
~~그건 아직 안 되더라.~~
```
- 틀렸던 예상, 셀프 디스, 농담처럼 끝낼 때

**밑줄 (`<u>텍스트</u>`)**
```markdown
이 부분이 <u>핵심</u>이다.
이 순간이 <u>바뀌는 순간</u>이었다.
```
- 굵게보다 더 강한 강조가 필요할 때
- 드물게 써야 효과 있다. 남용 금지.

**인라인 코드 (`` `코드` ``)**
```markdown
파일 맨 위에 `use client` 한 줄 추가하면 된다.
`npm run dev`로 실행한다.
```
- 명령어, 파일명, 함수명, 변수명
- writing 글에서도 터미널 명령어가 언급될 때는 쓴다

---

### 6. 목록

**순서 없는 목록 (`-`)**
```markdown
- 브라우저를 자동으로 연다
- 버튼을 클릭한다
- 결과를 확인한다
```
- 순서가 없는 나열. 항목이 3개 이상일 때 효과적
- 2개면 그냥 문장으로 쓰는 게 낫다

**순서 있는 목록 (`1.`)**
```markdown
1. 로봇이 브라우저를 연다
2. 글쓰기 페이지로 간다
3. 저장 버튼을 누른다
```
- 순서가 중요한 절차, 단계

**목록 주의사항:**
- 목록으로 쓰면 글이 설명서처럼 읽힌다. 남용 금지
- 목록 대신 문장으로 쓸 수 있으면 문장이 낫다
- writing 글에서는 목록이 거의 없어야 한다

---

### 7. 코드 블록 (dev 글 전용)

````markdown
```bash
npm run dev
```

```tsx
const slug = params.slug;
const post = await getPost(slug);
```
````

- **dev 글에서만.** writing 글에는 코드 블록 없다
- 에러 메시지, 명령어, 짧은 핵심 코드만
- 코드가 길어지면 설명이 묻힌다. "이런 역할을 하는 코드를 만들었다"는 말로 대체할 수 있으면 그게 낫다
- 언어 명시 필수: `js`, `ts`, `tsx`, `bash`, `css` 등

---

### 8. 표 (dev 글 전용)

```markdown
| 방법 | 속도 | 비고 |
|------|------|------|
| 직접 클릭 | 느림 | 매번 해야 함 |
| Playwright | 빠름 | 한 번 짜면 끝 |
```

- **dev 글에서만.** 비교하거나 정리할 게 있을 때
- 항목이 4개 이상일 때 표가 효과적. 3개 이하면 글로 써도 됨
- writing 글에서는 쓰지 않는다

---

### 9. 이미지 / GIF

```markdown
![캡션 텍스트](URL)    ← 캡션이 이미지 아래 표시됨
![](URL)               ← 캡션 없이
```

**GIF:**
- 글 하나에 **1~4개**. 그 이상이면 글 무게가 희석된다
- 아무 섹션에나 습관적으로 붙이면 안 된다. 감정이 크거나 분위기 전환이 필요한 순간에
- 기술 설명이 긴 글일수록 GIF 자제. GIF가 많으면 진지하게 읽히던 글이 가벼워진다

**이미지 (스크린샷 등):**
```markdown
![히어로 섹션 완성본 — 데스크탑](/images/blog/hero-desktop.png)
```
- 캡션이 있으면 이미지 아래에 표시됨
- 글 흐름을 방해하면 억지로 넣지 않는다

---

### 10. 구분선 (`---`)

```markdown
---
```

**렌더링**: `· · ·` 점 세 개로 표시됨.

**언제**: `##`보다 더 조용한 구분이 필요할 때. 분위기나 시간의 흐름이 바뀔 때.
**주의**: `##`이 있는 글에서 `---`까지 쓰면 과하다. 둘 중 하나만 선택.
- writing 글에서 감정의 전환점, 시간 이동에 잘 맞음
- dev 글에서는 `##`으로 구분하는 게 기본. `---`는 예외적으로만.

## 컴포넌트 선택 가이드

막힐 때 이 순서로 판단:

1. 핵심 한 줄이 있나? → `> !!` takeaway-box (글 전체에서 **1개**)
2. 기술 용어가 처음 나왔나? → `> 단어란?` def-box (최대 **3개**)
3. 중요한 인용이나 깨달음이 있나? → `> ` blockquote (1~2개)
4. 주제가 바뀌나? → `##` h2
5. 같은 성격의 항목이 3개 이상 나열되나? → 목록
6. 핵심 단어를 강조하고 싶나? → `**굵게**`
7. 속마음이나 혼잣말이 있나? → `*기울임*`
8. 틀렸던 생각이 있나? → `~~취소선~~`
9. 특히 강조하고 싶은 핵심 포인트? → `<u>밑줄</u>`
10. 코드/명령어가 나왔나? → `` `인라인 코드` `` 또는 코드 블록

## 글 타입별 자주 쓰는 패턴

### dev 글 전형적 구조
```markdown
# 제목

## 섹션 이름

본문. **핵심 강조**. *속마음*. ~~셀프 디스~~.

> 단어란? 설명 한 줄.

더 본문.

```bash
명령어
```

### 삽질 1: 구체적인 문제 이름

뭐가 안 됐는지 구체적으로...

> !! 핵심 한 줄 takeaway
```

### writing 글 전형적 구조
```markdown
# 제목

## 섹션 이름

본문. *속마음*. ~~셀프 디스~~.

---

## 다음 흐름 (또는 시간 이동)

더 본문.

> 감정이 응축된 한 문장.

더 본문.

> !! 결론이 있다면 여기. 없으면 생략.
```
