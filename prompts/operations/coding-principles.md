# Coding Principles

Lynn.ai는 육아 중에 유지보수되는 프로젝트다.
"지금 동작하는 것"보다 "나중에 뜯어고치지 않아도 되는 것"을 우선한다.

코드를 작성하기 전에 이 문서를 읽는다.
예외를 만들고 싶다면 이 문서를 다시 읽는다.

---

## 1. 중복을 만들지 않는다

작업 전에 "이미 이런 게 있지 않나?"를 먼저 확인할 것.

- 같은 로직, 같은 스타일, 같은 텍스트가 두 곳 이상에 있으면 반드시 공통화
- 한 곳을 고치면 전체에 반영되는 구조로 설계
- 복붙으로 해결하려는 충동 = 컴포넌트화하거나 추상화해야 한다는 신호

---

## 2. 하드코딩 금지

값, 색상, 간격, 문자열을 코드 안에 직접 박아 넣지 않는다.

```js
// ❌ 금지
<p style={{ fontSize: '14px', color: '#666' }}>...</p>
<h2 className="text-[22px] font-semibold">...</h2>

// ✅ 권장
<p className="text-body-sm text-muted">...</p>
<h2 className="text-heading-md">...</h2>
```

숫자, 색상, 반복 문자열은 토큰(CSS 변수, theme 설정, constants 파일)으로 정의 후 참조.

---

## 3. 텍스트 스타일은 정의된 것만 쓴다

새 스타일이 필요하면:
1. 기존 스타일 목록 먼저 확인
2. 정말 새로운 스타일이 필요한지 판단
3. 필요하다면 스타일 정의 파일에 추가
4. 추가한 스타일을 가져다 사용

즉흥적인 인라인 스타일 지정 금지. 스타일이 계속 늘어난다면 정리가 필요하다는 신호다.

---

## 4. UI 요소는 컴포넌트로 분리한다

- 2번 이상 쓰이면 반드시 컴포넌트화
- props로 변형 가능하게 설계 (variant, size 등)
- 파일명과 위치는 기존 구조와 일관되게

---

## 5. 가성비를 따지되 나중을 생각한다

빠른 방법이 떠올랐을 때 판단 기준:
- 기능 하나 더 붙을 때 이 구조가 버텨주는가?
- 6개월 후의 내가 이 코드를 이해할 수 있는가?
- 이 결정이 다른 부분에 불필요한 의존성을 만드는가?

하나라도 "아니오"라면 더 구조적인 방법을 먼저 찾는다.
"지금 빠르고 나중에 전부 뜯어야 하는" 방식은 선택하지 않는다.

---

## 6. 복잡해지면 리팩토링 먼저

새 기능 추가 전에 현재 코드 상태를 먼저 점검한다.

리팩토링이 필요한 신호:
- 하나의 파일/함수가 너무 많은 일을 하고 있다
- 같은 버그가 여러 곳에서 반복된다
- 새 기능 추가가 기존 코드 전체 이해를 요구한다

리팩토링과 기능 추가는 동시에 하지 않는다. 별도 작업으로 분리할 것.

---

## 7. 자체 테스트 후 Lynn에게 전달한다

작업 완료를 선언하기 전, 아래를 직접 확인한다.

**체크리스트:**
- [ ] 요청한 기능이 실제로 동작하는가?
- [ ] 기존에 되던 것이 깨지지 않았는가?
- [ ] 모바일/데스크탑 모두 확인했는가?
- [ ] 예상되는 엣지 케이스를 생각해봤는가?

예상 문제가 있다면 Lynn에게 보고 전에 코드를 한 번 더 검토한다.
문제를 발견했다면 수정 후 체크리스트를 처음부터 다시 돌린다.

Lynn에게 전달할 때 반드시 포함할 것:
1. 무엇을 했는지
2. 어떻게 테스트했는지
3. Lynn이 직접 확인해야 할 부분이 있다면 무엇인지

---

## 빠른 참조

| 상황 | 행동 |
|------|------|
| 비슷한 코드가 또 필요할 때 | 공통화 먼저 |
| 값을 직접 쓰고 싶을 때 | 토큰/상수로 정의 후 참조 |
| 새 스타일이 필요할 때 | 스타일 파일에 추가 후 사용 |
| UI 요소를 만들 때 | 컴포넌트로 분리 |
| 빠른 방법이 떠올랐을 때 | 나중에 뜯게 되는지 먼저 생각 |
| 코드가 복잡해졌을 때 | 기능 추가 전에 리팩토링 |
| 작업이 끝났을 때 | 체크리스트 돌리고 Lynn에게 전달 |
