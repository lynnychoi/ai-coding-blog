## 1. dev 글이란

- Claude Code로 뭔가를 만드는 과정을 기록하는 글
- 완성된 결과물 소개가 아니라 **과정과 삽질**이 중심
- 독자: 기획자, 디자이너, AI 코딩에 관심 있는 비개발자

## 2. 구조 원칙

### 한 가닥 흐름 (가장 중요)
글 전체에 **하나의 중심 질문 또는 갈등**이 있어야 한다. 오늘 한 것들을 나열하는 글이 아니다.

- 좋은 예: "비밀번호가 맞는데 왜 로그인이 안 됐는가" → 원인 탐색 → 해결 → 깨달음
- 나쁜 예: A 기능 만들고, B 기능 만들고, C도 만들고, 그래서 다 됐다

**포함할 거면 깊게, 뺄 거면 과감하게.**
- 두 가지 주제를 다 넣고 싶으면 두 편으로 쪼갠다
- 넣기로 했으면 표면만 긁지 말고, 독자가 "왜?"라고 물을 지점에 답이 있어야 한다

### 도입부 — 왜 이걸 했나
- "오늘은 ~를 만들어볼게요" 금지
- 어떤 불편함이나 문제가 있었는지에서 시작
- 독자가 "나도 그런 상황 있었는데" 하고 공감할 수 있게
- **작업 세션 맥락을 반드시 먼저 설명한다.** 이 작업이 왜 시작됐는지, 어떤 더 큰 흐름의 일부인지. 예를 들어 "dev server를 왜 띄우려고 했는지"조차 모르는 독자를 위해 — "수정한 글이 화면에 어떻게 보이는지 내 컴퓨터에서 직접 확인하려고 했다"는 식으로 목적부터 설명한다.

**왜 이게 중요한가**: 비개발자 독자는 "로컬 서버를 띄운다"는 행위 자체가 낯설다. 왜 그걸 하는지 설명 없이 기술적 증상으로 바로 들어가면 독자는 첫 문단에서 이미 길을 잃는다.

### 본문 — 과정 중심
반드시 포함해야 할 것:
- **뭐가 안 됐는지** — 막힌 지점을 구체적으로
- **왜 안 됐는지** — 이유를 설명하되 기술 용어는 바로 설명
- **어떻게 느꼈는지** — 황당함, 뿌듯함, 지침 등 감정도 기록

### 글 구조 (헤딩)
- `# 제목` 다음 바로 `## 첫 번째 섹션`으로 시작한다. 제목 아래에 바로 산문이 오면 안 됨
- `##` 없이 본문이 이어지면 회색 덩어리처럼 읽힌다. 최소 2~3개 섹션

### 마무리 — 핵심 하나
- 여러 교훈을 나열하지 않는다
- 하나만 골라서 인용구로:
  ```
  > 도구를 만든다는 건 내가 뭘 불편해하는지를 정확히 알아야 한다는 거다.
  ```

## 3. 기술 용어 처리 (핵심)

독자가 Next.js, API, 컴포넌트가 뭔지 모른다고 가정한다.

**용어 → def-box 순서 (절대 원칙)**

기술 용어는 반드시 **본문 문장에 먼저 등장**해야 한다. def-box는 그 용어가 쓰인 문장 바로 다음에만 온다.

```
❌ 잘못된 패턴:
"내 컴퓨터에서 임시 사이트를 띄워볼 수 있다."
> 로컬 서버란? 인터넷에 올리기 전에 내 컴퓨터에서만 돌아가는 임시 사이트다.
→ 본문에 "로컬 서버"라는 단어가 없으므로 def-box가 뜬금없이 떠있다.

✅ 맞는 패턴:
"**로컬 서버**를 켜서 먼저 확인할 수 있다."
> 로컬 서버란? 인터넷에 올리기 전에 내 컴퓨터에서만 돌아가는 임시 사이트다.
→ 본문에서 용어를 쓰고, def-box가 바로 설명한다.
```

- 설명은 한 줄. 길게 설명하지 않는다
- 비유를 쓰면 좋음 (일상적인 것으로)
- 모든 용어에 달 필요는 없다 — 흐름을 방해할 정도면 오히려 빼는 게 낫다
- 용어를 본문에서 빼고 def-box만 달면 독자는 맥락 없이 정의만 읽게 된다

**dev 글에서 자주 나오는 용어 — 반드시 설명 필요:**

| 용어 | 설명 예시 |
|------|----------|
| 로컬 서버 / dev server | 인터넷에 올리기 전에 내 컴퓨터에서만 돌아가는 임시 사이트 |
| localhost / 127.0.0.1 | "내 컴퓨터"를 가리키는 주소. 브라우저에서 이 주소로 들어가면 내 컴퓨터에서 돌아가는 사이트를 볼 수 있다 |
| 포트 (3000, 3011 등) | 컴퓨터의 출입구 번호. 같은 컴퓨터에서 여러 서버를 동시에 돌릴 때 번호로 구분한다 |
| 캐시 | 한번 불러온 것들을 저장해두는 임시 저장소. 다음에 빠르게 불러오려고 저장해두는 것 |
| 빌드 / 컴파일 | 내가 쓴 코드를 브라우저가 이해할 수 있는 형태로 변환하는 과정 |
| 터미널 | 코드로 컴퓨터에 명령을 내리는 검은 창 |
| 프로세스 | 컴퓨터에서 돌아가고 있는 프로그램 하나하나 |
| stderr / stdout | 프로그램이 메시지를 출력하는 두 가지 채널. stdout은 일반 메시지, stderr는 에러 메시지 |

이 중에서 글에 등장하는 것은 **처음 나올 때 반드시 설명한다.** 개발자한테 당연한 것도 독자에겐 낯선 단어다.

## 4. 코드 블록 사용 기준

- 코드는 **최소한으로** — 비개발자도 읽을 수 있어야 한다
- 코드 블록은 에러 메시지, 명령어, 짧은 핵심 코드만
- 긴 코드 설명 대신 "이런 역할을 하는 코드를 만들었다" 식으로

---

## 5. 삽질 기록 방식

삽질은 부끄러운 게 아니라 **좋은 소재**다.

- 틀렸던 생각: `~~처음엔 이렇게 하면 되는 줄 알았다~~`
- 막혔던 지점을 구체적으로 (막연하게 "힘들었다" 금지)
- 해결 과정도 중요하지만 해결 못 한 것도 솔직하게 써도 됨

---

## 6. dev 글 제목 패턴

제목은 글의 핵심 갈등이나 아이러니를 대표해야 한다. 무슨 기능을 만들었는지 요약하는 제목은 실패한 제목이다.

잘 맞는 패턴:
- `[기능/작업] 하는 데 [예상보다 오래 걸린 상황]` — 시간 대비 결과의 아이러니
  - "히어로 섹션 하나 만드는 데 하루 다 썼다"
- `[작업들] — [기간/상황] 요약` — 엄청난 양을 한 줄로
  - "댓글, 태그 필터, 읽기 시간 — 하루 만에 블로그 기능 다 붙인 날"
- `[무지함이나 아이러니 담은 상황]` — 핵심 긴장이 제목에
  - "보안이 뭔지도 모르면서 어드민 페이지 만든 날"
  - "비밀번호가 맞는데 왜 로그인이 안 됐냐"

**제목 테스트**: 제목만 봐도 "어, 이거 뭔 얘기야?" 하고 궁금해지면 성공. 활동 설명이면 실패.
