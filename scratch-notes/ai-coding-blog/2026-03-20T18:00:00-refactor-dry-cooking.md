# 코드 정리 — 같은 말을 두 번 하지 않기

## 프로젝트 한 줄 소개
Lynn의 개인 블로그. Claude Code로 기능을 하나씩 추가해오면서 어느 순간 같은 코드가 여러 파일에 흩어지기 시작했다.

## 이 작업을 하게 된 이유
블로그 기능이 어느 정도 갖춰지고 나서 전체 코드를 한번 돌아봤다.
아직 완성된 블로그는 아니지만, 지금 정리 안 하면 나중에 더 엉켜.
기능을 추가할 때마다 비슷한 코드를 복붙해왔는데, 이걸 하나로 합치는 게 리팩토링이다.

개발자가 아닌 입장에서 이해한 문제는 세 가지였다.

**1. 타입 중복**
`ImageEntry`(이미지 한 장의 정보 구조)가 글쓰기 페이지에도 정의돼있고, 글수정 페이지에도 똑같이 정의돼있었다.
레시피 양식이 두 부서에 각각 보관된 상황. 하나 바꾸면 다른 것도 바꿔야 하는데 까먹으면 따로 놀게 됨.
→ `app/cooking/types.ts` 하나로 통합.

**2. 유틸 함수 중복**
`today()`(오늘 날짜 반환 함수)가 API 파일 두 개에 각각 따로 있었다.
`commitImageToGitHub()`(이미지를 GitHub에 올리는 함수)도 두 파일에 복붙돼있었다.
→ `lib/utils.ts`와 `lib/github.ts`로 통합.

**3. 스타일 중복**
`#0a0a0f`(배경색), `#c4b8f8`(보라색) 같은 색상 코드가 여러 파일에 직접 박혀있었다.
버튼 스타일, 입력창 스타일도 파일마다 거의 비슷하게 반복.
→ `app/cooking/styles.ts` 하나로 통합.

## 뭘 했나 (작업 목록)
- `app/cooking/types.ts` 생성 — 타입 5개 통합 (ImageEntry, Status, SaveStatus, ActiveTab, EditorMode, GifResult, Fields, NoteItem)
- `lib/utils.ts` 생성 — today(), todayDatetime(), parseClaudeJson() 통합
- `lib/github.ts`에 `commitImageToGitHub` 추가 — generate-edit에 있던 중복 함수 흡수
- `app/cooking/styles.ts` 생성 — COLORS 팔레트 + 공통 스타일 상수
- write/page.tsx, edit/[slug]/page.tsx, API 라우트들에서 위 파일들 import로 교체

## 막혔던 것들 / 삽질
특별한 삽질은 없었다. 기계적인 작업이었음.
다만 스타일 파일을 만들 때 write/page.tsx의 `S`와 edit/page.tsx의 `DARK`가 비슷하지만 완전히 동일하지는 않아서
공통 부분만 추출하고 페이지별 차이는 각자 유지했다.

## 결정의 이유 (왜 이렇게 만들었나)
- 아직 블로그 기능이 완성되지 않아서 과도한 추상화는 피했다. 딱 중복되는 것만 뺐다.
- 커스텀 훅이나 컴포넌트 분리는 나중에. 지금은 타입/유틸/스타일 3가지만.
- `lib/`은 서버/클라이언트 공용, `app/cooking/`은 프론트엔드 전용으로 위치 구분.

## 그때 어떤 기분이었나
솔직히 리팩토링은 티가 잘 안 나서 뿌듯하기가 애매하다.
기능이 새로 생긴 것도 아니고 디자인이 바뀐 것도 아니니까.
근데 코드를 읽을 때 훨씬 깔끔하다는 건 느껴진다.
나중에 색상 바꾸거나 타입 수정할 때 한 군데만 고쳐도 된다는 게 편할 것 같다.

## 배운 것 / Takeaway
- 기능을 빠르게 추가하다 보면 자연스럽게 중복이 생긴다. 나쁜 게 아니라 그냥 그렇게 됨.
- 리팩토링은 코드를 예쁘게 만드는 게 아니라 나중에 수정할 때 한 곳만 고치게 만드는 것.
- 개발자가 아니어도 "같은 말을 두 번 하지 마라"는 원칙은 이해할 수 있다.
- **핵심 한 줄**: 같은 코드가 두 군데 있으면, 고칠 때 두 군데를 다 찾아야 한다. 그게 싫으면 지금 합쳐라.
