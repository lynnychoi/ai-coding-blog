## 프로젝트 한 줄 소개

Lynn의 개인 블로그. Claude API가 글을 마크다운으로 변환해서 GitHub에 커밋하는 구조. Vercel 배포. 어드민(cooking)에서 글 쓰기·편집·관리. — 어느 정도 안정됐다고 생각했는데 하루에 세 개의 문제가 동시에 터졌다.

## 이 작업을 하게 된 이유

같은 날 세 가지가 한 번에 거슬렸다:

1. **빈 응답 버그** — 글 생성 누르면 "서버 오류 (빈 응답)"만 뜨고 아무것도 안 됨. 매번.
2. **비공개 안 되는 버그** — "비공개"로 설정해도 항상 공개로 올라감.
3. **어드민 UX** — Cooking Station 플로팅 버튼이 모바일에서 콘텐츠를 가림. 목록에서 새 글 쓰러 가는 동선도 너무 김.

세 개 모두 한 날에 해결.

## 뭘 했나

### 1. 빈 응답 버그 — 3연전

#### 1라운드: Giphy 타임아웃
- Vercel Hobby 플랜은 서버리스 함수에 60초 제한.
- 글 생성하면서 Giphy API까지 호출하다가 시간 초과.
- "서버 오류" 메시지만 보여 원인 파악 어려움.
- → **Giphy 자동 삽입 기능을 통째로 제거** (`/api/write/route.ts`). 어차피 GIF는 직접 찾아 넣는 게 더 낫기도 했고.

#### 2라운드: 로그 쓰기 실패 + try-catch 부재
- Giphy 제거했는데도 실패.
- `logs/generation-log/` 폴더에 생성 기록 저장하는 코드 — Vercel 서버리스는 **파일시스템이 읽기 전용**.
- 로컬에선 잘 되니까 몰랐던 것. 예외 터졌고 잡는 코드 없어서 서버 함수가 그냥 죽음.
- → POST 핸들러 전체를 outer try-catch로 감쌈. 로그 쓰기도 try-catch 처리해서 실패해도 무시.

#### 3라운드: max_tokens 잘림
- 속도 개선하려고 `max_tokens`를 8192 → 4096으로 줄였는데, 블로그 글은 4096을 가볍게 넘김.
- Claude가 JSON 응답을 중간에 잘랐고 파싱 실패.
- → max_tokens 8192로 복원. 속도보다 완성도.

### 2. 비공개 안 되는 버그 — 코드가 아니라 프롬프트 문제

- 처음엔 `route.ts`를 뒤졌다. `formData.get("status")`로 Claude한테 잘 넘기고 있었다. 코드는 멀쩡.
- 진짜 원인: `LYNN-BLOG-PRINCIPLES.md`의 frontmatter 출력 템플릿에 **`status` 필드 자체가 없었다.**
  ```
  # 잘못된 템플릿
  title: "제목"
  date: YYYY-MM-DD
  tags: ["태그1"]
  type: dev
  ```
- Claude는 시키는 대로만 한다. 템플릿에 없으면 안 쓴다. `status` 필드 없는 마크다운 → `lib/posts.ts`가 기본값으로 공개 처리.
- → 템플릿에 `status: unpublished` 추가. 즉시 해결.

**부가**: 글 작성 페이지 기본값을 `published` → `unpublished`로. 실수로 공개되는 것보다 실수로 비공개되는 게 낫다.

**부가 2**: 날짜 입력 필드 추가. scratch-note 불러올 때 노트 날짜가 자동 입력되게 (며칠 전 작업을 오늘 글로 써도 발행일이 맞게).

### 3. 어드민 UX — Cooking Station 접근 동선

## 막혔던 것들 / 삽질

### 빈 응답 — 두더지 게임
같은 에러가 계속 나는데 원인이 매번 달랐다. 하나 고치면 다른 에러가 나오는 식. **에러 메시지가 "빈 응답"이라 아무 정보도 안 줬다.** Vercel 대시보드까지 들어가서 로그 봐야 했는데 그것도 귀찮음. 처음부터 outer try-catch로 잡았어야 했다.

### 비공개 버그 — 코드가 아니라 프롬프트
처음엔 내가 버튼을 잘못 누른 줄 알았다. 몇 번 반복되니까 버그라는 걸 알았고, 코드 뒤지다 보니 **AI에게 지시하는 문서에 빠진 한 줄이 실제 동작을 바꿔놨다는 게 신기했다.** 코드 버그보다 "프롬프트 버그"가 더 찾기 어려운 것 같다.

### hover 편집 버튼 — 만들었다 지운 이유
목록 카드에 hover 시 편집 버튼이 나타나게 하는 계획이었다. 구현은 됐는데:
- 태그·읽기 시간 텍스트 위에 버튼이 겹쳐 보임
- CSS로 위치 잡으려 하니 카드 레이아웃이 글마다 달랐다 (태그 있는/없는, 태그 많은)
- "오른쪽 고정"으로 바꿨더니 모바일에서 또 다른 문제
- 고치다 보니 원래 코드보다 복잡해지고 있음

그 시점에 멈추고 생각: ***이 버튼이 정말 필요한가?*** Cooking Station에서 글 목록 보고 편집하는 게 이미 된다. 목록 페이지에 편집 버튼까지 있을 필요가 있나? → 없다. **지웠다.**

### 클라이언트/서버 컴포넌트
`AdminListControls`는 로그인 상태 확인하려고 `/api/auth/status` fetch 필요 = 클라이언트 컴포넌트. 사용처(`app/blog/page.tsx`)는 서버 컴포넌트. App Router에서는 서버 → 클라이언트 nesting만 가능. `"use client"` + `useEffect`로 처리. 로딩 중 깜빡임 막으려 `useState` 초기값 `false`로.

## 결정의 이유

- **Giphy 완전 제거** — 자동 삽입보다 직접 고르는 게 글 품질에 낫다. 타임아웃도 해결. 일석이조.
- **로그 쓰기 try-catch** — Vercel에선 안 되는 기능. 로컬 전용으로 남기되 실패 시 조용히 넘어가게.
- **max_tokens 8192 유지** — 속도보다 완성도. 잘린 글이 나오면 의미 없다.
- **기본값을 `unpublished`로** — 실수로 숨겨지는 게 실수로 노출되는 것보다 훨씬 낫다.
- **플로팅 버튼 제거** — nav에 링크 생겼으니 중복. 플로팅은 콘텐츠 가림.
- **편집 버튼은 목록이 아닌 글 상세에만** — 글 읽다가 수정하고 싶을 때 필요한 거지, 목록에서 편집할 일은 드물다.
- **새 글 쓰기 버튼은 목록에** — 목록 보다가 "나도 써야지" 흐름이 자연스럽다.

## 그때 어떤 기분이었나

빈 응답 버그는 **두더지 게임 같았다.** 하나 고치면 또 다른 에러. 세 번째 라운드에서 max_tokens 문제 발견했을 땐 좀 웃겼다. 너무 당연한 원인을 왜 처음부터 못 봤지? 빠르게 만들려다 더 오래 걸린 셈.

비공개 버그는 원인 찾고 좀 황당했다. **코드가 멀쩡한데 프롬프트가 잘못된 거였다.** AI 사용하는 시스템에선 코드 밖에서도 버그가 생긴다. 새로운 종류의 버그를 만난 느낌.

hover 편집 버튼 만들었다 지울 때 좀 아까웠다. 시간을 들였는데 결국 없애는 결정. "이게 필요한가?"라는 질문을 **코딩 시작 전**에 했어야 하는데 만들고 나서 했다. **기획자 출신인데 정작 내 프로젝트에선 기획을 대충 하는 아이러니.**

nav에 🍳 이모지 하나로 Cooking 링크를 만든 건 마음에 든다. 작고 귀엽고, 로그인한 사람만 보인다. 뭔가 비밀스러운 느낌.

## 배운 것 / Takeaway

- **에러 메시지가 없는 에러가 제일 무섭다** — "빈 응답"은 정보 없음. 처음부터 outer try-catch로 모든 에러를 응답에 포함시켰어야.
- **환경 차이를 항상 의심하라** — 로컬에선 되는데 배포에선 안 되는 건 보통 파일시스템, 환경변수, 타임아웃.
- **성능 최적화는 동작 확인 후에** — max_tokens 줄여 속도 올리려다 버그만 만들었다.
- **Claude는 시키는 대로만 한다** — 템플릿에 없으면 안 쓴다. 프롬프트가 코드만큼 중요.
- **기본값은 항상 "안전한 방향"** — 공개/비공개라면 비공개가 기본이어야.
- **버그를 코드에서만 찾지 마라** — 프롬프트 버그도 있다.
- **만들기 전에 "정말 필요한가?" 먼저 물어라** — 만들고 나서 묻기 시작하면 늦다.
- **복잡해지기 시작하면 멈추고 다시 생각** — 복잡해진다는 건 방향이 잘못됐다는 신호일 수 있다.
- **삭제도 작업이다** — 만들었다 지운 코드도 판단을 한 거다. 부끄러운 게 아니라.
- **핵심 한 줄**: 에러가 없는 게 아니라 에러가 숨어 있었고, 옵션이 무시된 게 아니라 옵션이 전달되지 않았고, 만든 게 아까운 게 아니라 안 만들었어도 됐던 거다.
