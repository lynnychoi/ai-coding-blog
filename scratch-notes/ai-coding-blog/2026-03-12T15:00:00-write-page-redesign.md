# 새 글 쓰기 페이지 개선 + UI 통일

## 프로젝트 한 줄 소개
Claude Code로 만드는 개인 블로그. 기획자 출신 Lynn이 AI 코딩 과정을 기록. 모든 도구는 내가 직접 쓰는 용도라 UX가 실제로 편해야 한다.

## 이 작업을 하게 된 이유
두 가지 문제가 생겼다.

첫째, 새 글 쓰기 페이지에 메모 하나만 입력하는 칸이 있었는데, "내용(재료)"과 "Claude에게 하는 지시"가 섞여서 들어갔다. 나중에 편집 페이지에서 원본 메모를 보려고 하면 "어디까지가 내 생각이고 어디까지가 Claude 지시였는지" 구분이 안 됐다.

둘째, 새 글 쓰기 페이지와 편집 페이지의 생김새가 너무 달랐다. 새 글 쓰기는 초록빛 계열, 편집은 보라 계열, 레이아웃도 달라서 왔다갔다 하면 헷갈렸다. 같은 블로그 어드민인데 앱이 두 개인 느낌.

## 뭘 했나
- **새 글 쓰기 페이지 재구성**:
  - "재료" 칸 — 오늘 뭐 했는지, 키워드, 메모. 이 내용은 frontmatter `notes`에 저장됨. 나중에 편집 페이지에서 볼 수 있음.
  - "Claude에게" 칸 — 특별 지시사항. 글 생성할 때만 쓰이고 저장 안 됨. 배경색도 다르게 해서 시각적으로 구분.
- **`app/api/write/route.ts`** — `prompt` 파라미터 추가. Claude 지시에 "추가 지시" 섹션으로 포함.
- **`prompts/blog-writing-rules.md`** — 출력 형식에 `notes` 필드 추가. Claude가 글 쓸 때 원본 메모를 frontmatter에 그대로 넣도록.
- **마크다운 미리보기 추가**: 편집 페이지에 편집/미리보기 토글. `react-markdown` 사용 (이미 설치되어 있었음). h1~h3 크기 차별화, blockquote, 코드블록, 이미지 스타일 적용.
- **UI 통일**: 두 페이지 모두 상단 고정 바, 섹션 구분선, 작은 대문자 라벨, 보라 계열(#c4b8f8) 통일.
- **scratch-notes 시스템 구축**: `scratch-notes/` 폴더 생성. `HOW-TO-WRITE.md` 작성 원칙 파일 만들고, 오늘 작업들 소급해서 기록.

## 막혔던 것들 / 삽질
- **미리보기에서 GIF가 안 보임**: 실제 Giphy URL은 잘 불러오는데, `react-markdown`이 이미지를 렌더링할 때 Next.js `Image` 컴포넌트가 아니라 `<img>` 태그를 써서 경고가 떴다. `eslint-disable` 주석으로 처리.
- **재료/지시 분리 방향 결정**: "저장되는 것"과 "저장 안 되는 것"을 UI에서 명확히 구분해야 사용자(나)가 혼동 안 한다. 배경색을 다르게 한 것도 그 이유.

## 결정의 이유
- 재료(notes)를 저장하고 지시(prompt)는 저장 안 하는 구분: 나중에 같은 재료로 다시 글을 쓰거나 다른 방향으로 수정할 때, "내가 그날 실제로 뭘 했는지"가 남아 있어야 한다. Claude 지시는 그때그때 달라지는 것.
- UI 통일: 자주 쓰는 도구일수록 일관성이 중요하다. 버튼이 어디 있는지 생각 안 해도 손이 가야 한다.

## 그때 어떤 기분이었나
두 페이지가 통일되고 나니까 훨씬 편했다. "아 이거 어드민이구나" 하는 느낌이 생겼다. scratch-notes 시스템을 만드는 건 좀 메타적인 작업이었는데, "나중에 이게 블로그 글이 된다"는 생각을 하니 기록을 꼼꼼히 해야겠다는 동기부여가 됐다. 지금 이 파일 쓰는 것처럼.

## 배운 것 / Takeaway
- "재료"와 "지시"는 다르다. 재료는 사실이고 지시는 관점이다. 섞이면 나중에 구분이 안 된다
- UI 일관성은 기능 추가만큼 중요하다. 헷갈리는 도구는 안 쓰게 된다
- 작업 기록을 남기는 것 자체가 작업이다. 그냥 지나치면 나중에 블로그 쓸 때 "내가 왜 이렇게 했더라" 하게 됨
- **핵심 한 줄**: 나를 위해 만드는 도구는, 내가 실제로 쓰고 싶어야 잘 만든 거다.
