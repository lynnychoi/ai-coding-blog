## 프로젝트 한 줄 소개

Lynn의 개인 블로그. 어드민(cooking)에서 글을 쓰고 편집한다. 3월 12일 주말에 어드민 시스템 뼈대를 다 만들었고, 일주일쯤 굴려보니 자잘한 버그와 UX 문제가 보이기 시작했다.

## 이 작업을 하게 된 이유

기능은 일단 다 붙였는데 실제로 쓰다 보니 세 가지가 거슬렸다:

1. **버그** — 비공개로 저장해도 대시보드 "비공개" 필터에 안 걸린다
2. **UX** — 모바일에서 타입·공개여부 드롭다운이 너무 번거롭다
3. **구조** — write 페이지와 edit 페이지에 같은 코드가 중복됨. 앞으로 기능 추가하려면 두 곳 다 손대야 함

같은 날 셋 다 해결하기로 했다. "기능 다 붙이고 나서 정리하는 날."

## 뭘 했나

### 1. 버그픽스 — status 필터 안 걸리는 문제

- **원인**: `commitToGitHub()`은 GitHub에만 쓰는데, `getAllPosts()`는 로컬 `content/posts/` 파일을 읽음. 로컬은 그대로니까 필터가 stale 데이터를 보고 있었다.
- **해결**: `PUT /api/admin/posts/[slug]` 핸들러에 dev 환경에서 로컬 파일도 동기화하는 코드 추가.
  ```ts
  if (process.env.NODE_ENV === "development") {
    const localPath = path.join(process.cwd(), "content", "posts", `${slug}.md`);
    fs.writeFileSync(localPath, markdown, "utf-8");
  }
  ```
- `POST /api/write`에도 동일 적용 — 신규 글 생성 후 대시보드에 즉시 반영.

### 2. UX 개선 — 드롭다운을 토글 버튼으로

- 기존: `<select>` 드롭다운 → 모바일에서 불편, 줄줄이 나열
- 변경: 탭 형태 토글 버튼
  - **타입**: `dev` (보라) / `writing` (회색)
  - **공개여부**: `공개` (초록) / `비공개` (보라)
- 날짜 + 타입 + 공개여부 **한 줄**에 배치
  - 데스크탑: 넓게 잘 보임
  - 모바일(390px): "비공개" 글자 12px로 줄여서 한 줄 안에 들어옴
- 파일: `app/cooking/edit/[slug]/page.tsx`, `app/cooking/write/page.tsx`

### 3. 리팩토링 — write/edit 통합 아키텍처

write/edit 페이지에 동일 코드가 두 벌씩 있었다. 앞으로 기능 추가할 때마다 양쪽 다 수정해야 하는 구조가 문제.

**아키텍처 결정:**
```
/cooking/write      → 얇은 입력 폼 (재료, 프롬프트, 태그, 타입, 이미지)
                      생성 완료 시 → /cooking/edit/[slug]?new=1 로 리다이렉트

/cooking/edit/[slug] → 모든 편집 기능이 있는 단일 에디터
                       신규/기존 글 모두 여기서 처리
```

앞으로 새 기능은 edit 페이지에만 추가하면 됨.

**공통 컴포넌트 신설** (`app/cooking/components/`):
- `TypeStatusRow.tsx` — 타입 + 공개여부 토글 한 줄
- `TagsInput.tsx` — 쉼표 구분 태그 입력
- `ImageUploadList.tsx` — 이미지 미리보기 + 설명 + 삭제

**write 페이지 변경:**
- 성공 화면(done state) 제거
- 생성 후 `router.push('/cooking/edit/${slug}?new=1')` 리다이렉트
- 버튼 텍스트: "Claude한테 올려줘 →" → "Claude한테 초안 만들어줘 →"
- 하단 안내: "생성 후 편집 페이지에서 바로 수정할 수 있어"

**edit 페이지 변경:**
- `useSearchParams()`로 `?new=1` 감지
- `isNew`면 상단 초록 배너:
  "✦ 초안이 생성됐어. 여기서 바로 수정하거나 AI한테 더 요청해봐. 만족하면 저장하기."

## 막혔던 것들 / 삽질

- **버그 원인 찾는 데 시간 들었다** — 처음엔 코드 로직이 틀린 줄 알았다. dev 환경에서 GitHub와 로컬이 다른 소스로 작동하고 있다는 사실을 깨닫는 데 시간이 걸림.
- **구조 변경 후 흐름 깨질까봐 조마조마** — write → edit 리다이렉트는 좋은 결정인데, 사용자(나)가 처음엔 "어 저장됐다는 표시가 없는데?" 할 수 있어서 isNew 배너를 명확하게 표시해야 했다.

## 결정의 이유

- **dev 환경 로컬 동기화** — GitHub만 보는 게 맞지만 dev에서 즉시 반영이 안 되면 디버깅이 너무 번거롭다. 환경 변수로 분리.
- **드롭다운 대신 토글** — 모바일이 주 사용 환경. 한 번 탭으로 선택 가능한 게 빠르다.
- **edit 페이지를 단일 에디터로** — write는 진입점만, 편집의 본진은 edit. 그래야 기능 추가가 한 곳에만 모인다.
- **공통 컴포넌트 분리** — 같은 UI 패턴(타입 토글, 태그 입력, 이미지 리스트)이 두 페이지에 반복되면 무조건 한쪽이 더 발전하고 나머지가 뒤처진다.

## 그때 어떤 기분이었나

버그를 발견했을 때 처음엔 "내가 잘못 눌렀나?" 했다가, 몇 번 반복되니까 "이건 진짜 버그구나" 깨달았다. 그러고 보면 사용자가 "내가 실수한 줄 알았다"고 말하기 전에 시스템을 의심하기 시작하는 순간이 있다. 그 순간이 디자이너로서 미안한 순간.

리팩토링하면서는 좀 시원했다. 같은 코드가 두 곳에 있는 게 늘 거슬렸는데, 정리하고 나니 다음에 기능 추가할 때 한 곳만 보면 된다는 게 명확해졌다.

토글 버튼으로 바꾸고 나서 모바일에서 한 번 써봤는데 — 진짜 빨라졌다. 작은 변화인데 매번 쓰는 거라 차이가 크다.

## 배운 것 / Takeaway

- **같은 데이터에 두 소스가 있으면 한쪽이 항상 stale이다** — dev에서 GitHub과 로컬을 분리하는 건 편리하지만 동기화 빼먹으면 바로 버그.
- **모바일이 주 사용 환경이면 모든 인터랙션을 거기 기준으로 짜야 한다** — 데스크탑에선 드롭다운이 멀쩡해 보여도 모바일에선 손가락 동선이 길어진다.
- **중복 코드는 시간이 지날수록 비싸진다** — 처음엔 복붙이 빠른데, 기능 추가하면서 한쪽만 고치면 회복 불가능한 부채가 된다.
- **사용자가 "내가 잘못한 거 같아"라고 생각하는 순간을 만들지 마라** — 버그가 사용자의 자신감을 깎는다.
- **핵심 한 줄**: 기능을 다 붙인 다음 정리하는 시간을 따로 잡지 않으면, 그 시점은 영원히 오지 않는다.
