# 비공개로 설정했는데 왜 공개로 올라가? — frontmatter 필드 버그

## 프로젝트 한 줄 소개
Lynn의 개인 블로그. 글을 쓸 때 Claude가 마크다운 파일을 생성하고 GitHub에 커밋함. 각 마크다운 파일 상단 frontmatter에 제목, 날짜, 태그, 공개 여부 등이 저장됨.

## 이 작업을 하게 된 이유
글 작성 페이지에서 "비공개"를 선택해도 글이 항상 공개로 올라갔다. 처음 한두 번은 내가 잘못 누른 줄 알았다. 근데 계속 똑같은 일이 반복됐다. 임시 글, 아직 덜 다듬은 글이 그냥 공개로 올라오는 거라 좀 민망했다.

## 뭘 했나 (작업 목록)
- `LYNN-BLOG-PRINCIPLES.md` frontmatter 템플릿에 `status: unpublished` 필드 추가
- `app/cooking/write/page.tsx` 기본값을 `published`에서 `unpublished`로 변경
- 날짜 입력 필드 추가 (오늘 날짜 기본값, 직접 수정 가능)
- scratch-note 불러올 때 해당 노트의 날짜가 자동으로 반영되게 수정

## 막혔던 것들 / 삽질

### 비공개 설정이 작동하지 않았던 진짜 이유
처음엔 API 코드 어딘가에서 status를 무시하는 부분이 있나 싶어서 `route.ts`를 뒤졌다. 근데 코드 자체는 문제가 없었다. `formData.get("status")`를 읽어서 Claude에게 넘기고 있었다.

문제는 Claude였다. 정확히는 Claude에게 주는 글쓰기 원칙 파일(`LYNN-BLOG-PRINCIPLES.md`)에 있는 frontmatter 출력 템플릿이었다. 그 템플릿에 `status` 필드 자체가 없었다.

```
# 잘못된 템플릿 (status 없음)
title: "제목"
date: YYYY-MM-DD
tags: ["태그1"]
type: dev
```

Claude는 시키는 대로 한다. 템플릿에 없으면 쓰지 않는다. 그러면 마크다운 파일에 `status` 필드 자체가 없는 상태로 저장된다. 그 상태에서 블로그 코드(`lib/posts.ts`)는 status 필드가 없으면 기본값으로 공개 처리했다.

→ 결국 내가 "비공개"를 눌렀어도, Claude가 생성한 파일에 status 필드가 없었고, 없으면 공개가 되는 구조였다.

템플릿에 `status: unpublished`를 추가하니까 바로 해결됐다.

### 날짜도 이상했다
글 작성 날짜가 scratch-note에 기록된 날짜와 달랐다. 며칠 전 작업을 오늘 글로 쓰면 발행일이 오늘로 찍혔다.

날짜 입력 필드를 추가하고, scratch-note를 불러올 때 그 노트의 날짜가 자동으로 입력 필드에 채워지게 했다. 필요하면 직접 수정도 가능.

## 결정의 이유
- 기본값을 `unpublished`로 바꾼 건: 글 쓸 때마다 직접 비공개로 바꾸는 게 번거롭고, 실수로 공개가 되는 것보다 실수로 비공개가 되는 게 훨씬 낫다.
- 날짜 입력 필드: 오래된 메모를 나중에 글로 쓸 때 발행일을 맞춰야 하는 경우가 많다.

## 그때 어떤 기분이었나
처음엔 내가 실수한 줄 알았다. 버튼을 잘못 눌렀나, 다시 확인해야지, 하면서 몇 번을 다시 설정했다. 근데 계속 공개로 올라오니까 이건 내 실수가 아니라 버그라는 걸 알았다.

원인을 찾고 나서는 좀 황당했다. 코드 로직의 문제가 아니라 프롬프트 템플릿에 필드가 없어서 생긴 버그였다. AI한테 지시하는 문서에 빠진 한 줄이 실제 동작을 바꿔놨다는 게 신기했다. 코드 버그보다 "프롬프트 버그"가 더 찾기 어려운 것 같다.

## 배운 것 / Takeaway
- **Claude는 시키는 대로만 한다.** 템플릿에 없으면 쓰지 않는다. 프롬프트가 코드만큼 중요하다.
- **기본값은 항상 "안전한 방향"으로.** 공개/비공개라면 비공개가 기본값이어야 한다. 실수로 숨겨지는 게 실수로 노출되는 것보다 낫다.
- **버그를 코드에서만 찾지 말 것.** 이번 버그는 프롬프트 문서에 있었다. AI를 사용하는 시스템은 코드 밖에서도 버그가 생긴다.
- **핵심 한 줄**: 옵션을 설정했어도 그 옵션이 전달되지 않으면 아무 의미가 없다.
