# 만들었다 지웠다 — 어드민 UX 삽질기

## 프로젝트 한 줄 소개
Lynn의 개인 블로그. 로그인한 상태에서는 글 관리 도구들(Cooking Station)이 보여야 한다. 로그아웃 상태에서는 일반 독자처럼 보여야 하고.

## 이 작업을 하게 된 이유
Cooking Station(어드민 페이지) 링크가 너무 불편한 곳에 있었다. 화면 한쪽에 떠있는 플로팅 버튼으로만 접근할 수 있었는데, 모바일에서 글을 읽다 보면 이 버튼이 콘텐츠를 가렸다. 그리고 Dev Log나 Writing 목록 페이지에서 새 글을 쓰려면 일부러 Cooking Station까지 들어갔다 나와야 했다. 동선이 너무 길었다.

"로그인 상태면 바로 접근할 수 있게 해주자"는 단순한 목표로 시작했는데, 이걸 구현하면서 예상보다 많이 만들었다 지웠다를 반복했다.

## 뭘 했나 (작업 목록)
- `StickyHeader.tsx`: 로그인 상태에서 nav에 🍳 아이콘으로 Cooking Station 링크 추가
- `AdminBar.tsx`: 플로팅 Cooking Station 버튼 완전 제거. 이제 글 상세 페이지에서만, 그것도 slug가 있을 때만 편집 링크 표시
- `AdminListControls.tsx` 신규 컴포넌트 생성: 로그인 상태에서 목록 페이지 하단에 "새 글 쓰기" 버튼 표시
- Dev Log 페이지, Writing 페이지에 `AdminListControls` 추가
- ~~각 글 카드에 hover 시 편집 링크 추가~~ → 만들었다가 지움

## 막혔던 것들 / 삽질

### hover 편집 버튼: 만들었다 지운 이유
처음 계획은 목록에서 각 글 카드에 마우스를 올리면 편집 버튼이 나타나게 하는 거였다. 구현은 됐는데 문제가 있었다.

태그랑 읽기 시간 텍스트 위에 편집 버튼이 겹쳐 보였다. CSS로 위치를 잡으려 했는데, 카드 레이아웃이 글마다 달라서 (태그 없는 글, 태그 많은 글) 위치가 제각각이었다. "가장 오른쪽에 고정으로"로 다시 짜봤는데, 모바일에서는 또 다른 문제가 생겼다.

고치다 보니 원래 코드보다 복잡해지고 있었다. 그 시점에 잠깐 멈추고 생각해봤다: *이 버튼이 정말 필요한가?* Cooking Station에서 글 목록 보고 편집하는 게 이미 된다. 목록 페이지에서 편집 버튼까지 있을 필요가 있나?

결론: 필요 없다. 지웠다.

### 클라이언트 컴포넌트와 서버 컴포넌트 문제
`AdminListControls`를 만들 때, 로그인 상태를 확인하려면 `/api/auth/status`를 fetch해야 한다. 이건 브라우저에서 실행되어야 한다. 즉 클라이언트 컴포넌트여야 한다.

근데 이걸 사용하는 `app/blog/page.tsx`는 서버 컴포넌트다. Next.js App Router에서는 서버 컴포넌트 안에 클라이언트 컴포넌트를 넣을 수 있다. 반대는 안 된다. 그래서 `"use client"` 지시어를 파일 맨 위에 명시하고 `useEffect`로 auth 상태를 가져오게 했다.

로그인 안 한 상태에서는 버튼이 아예 렌더링되지 않아야 했는데, 처음엔 로딩 중에 버튼이 잠깐 보였다 사라지는 깜빡임이 있었다. `useState`의 초기값을 `false`로 해서 해결했다.

## 결정의 이유
- **플로팅 버튼 제거**: nav에 링크가 생겼으니 이제 필요없다. 두 곳에 같은 링크가 있을 필요 없고, 플로팅 버튼은 콘텐츠를 가린다.
- **편집 버튼은 목록이 아닌 글 상세에만**: 목록에서 편집할 일은 거의 없다. 글 읽다가 수정하고 싶을 때 편집 버튼이 필요하다.
- **새 글 쓰기 버튼은 목록에**: 글 목록 보다가 "나도 써야지" 하는 흐름이 자연스럽다.

## 그때 어떤 기분이었나
편집 버튼 만들었다 지울 때 좀 아까웠다. 시간을 들여 만들었는데 결국 없애는 결정을 했으니까. 근데 억지로 유지하는 것보다 낫다는 걸 알고 있었다.

"이게 필요한가?"라는 질문을 코딩 시작 전에 했어야 하는데, 만들고 나서 하게 됐다. 기획자 출신인데 정작 내 프로젝트에서는 기획을 대충 하는 아이러니.

nav에 🍳 이모지 하나로 Cooking Station 링크를 만든 건 마음에 든다. 작고 귀엽고, 로그인한 사람만 보인다. 뭔가 비밀스러운 느낌.

## 배운 것 / Takeaway
- **만들기 전에 "정말 필요한가?"를 먼저 물어야 한다.** 기획자 출신이라면서 정작 내 개발에서는 그 습관이 빠졌다. 코딩이 재밌으면 일단 만들게 되는데, 그게 항상 좋은 건 아니다.
- **복잡해지기 시작하면 멈추고 다시 생각하자.** CSS 위치 잡는 게 점점 복잡해졌을 때 멈췄어야 했다. 복잡해진다는 건 방향이 잘못됐다는 신호일 수 있다.
- **삭제도 작업이다.** 만들었다 지운 코드도 판단을 한 거다. 부끄러운 게 아니라.
- **핵심 한 줄**: 기획자 출신인데 기획을 코딩 후에 했다.
