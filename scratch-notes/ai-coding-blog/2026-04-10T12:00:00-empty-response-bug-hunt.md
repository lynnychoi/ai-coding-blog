# 글이 왜 안 써지지 — 빈 응답 버그 3연전

## 프로젝트 한 줄 소개
Lynn의 개인 블로그. Claude Code로 직접 개발 중. 글을 쓰면 Claude API가 마크다운으로 변환해서 GitHub에 커밋하는 구조. Vercel에 배포되어 있음.

## 이 작업을 하게 된 이유
블로그 글 생성이 자꾸 실패했다. "서버 오류 (빈 응답)"이라는 메시지만 뜨고 아무것도 안 됐다. 처음엔 그냥 일시적인 문제겠지 싶었는데 매번 똑같이 실패했다. 블로그 시스템 자체가 핵심 기능인데 글이 안 써지면 아무 의미가 없으니 반드시 고쳐야 했다.

## 뭘 했나 (작업 목록)
- Giphy 자동 삽입 기능 제거 (`/api/write/route.ts`)
- POST 핸들러 전체를 outer try-catch로 감쌈
- 로그 파일 쓰기 부분도 try-catch 처리 (Vercel 읽기전용 파일시스템 대응)
- max_tokens 4096으로 줄였다가 → 파싱 오류 발생 → 다시 8192로 복원
- parseClaudeJson 함수가 잘린 JSON 처리 못하는 것도 확인

## 막혔던 것들 / 삽질

### 1라운드: Giphy 타임아웃
Vercel Hobby 플랜은 서버리스 함수에 60초 제한이 있다. 글 생성하면서 Giphy API까지 호출하다 보니 시간 초과가 났던 것. 근데 처음엔 이게 원인인 줄 몰랐다. "서버 오류"라는 메시지뿐이고 어떤 에러인지 알 수가 없었다. 에러 로그를 봐야 했는데 Vercel 대시보드까지 들어가서 확인하기가 귀찮았다.

결국 Giphy 자동 삽입 기능을 통째로 들어냈다. 어차피 원하는 GIF를 직접 찾아서 넣는 게 더 낫기도 했고.

### 2라운드: 빈 응답의 진짜 원인
Giphy를 없앴는데도 여전히 실패했다. 이번엔 다른 이유였다.

`logs/generation-log/` 폴더에 생성 기록을 저장하는 코드가 있었는데, Vercel 서버리스 환경은 파일시스템이 읽기전용이다. 로컬에서는 문제없이 동작하니까 몰랐던 것. 이 코드에서 예외가 터졌고, 그 예외를 잡는 코드가 없었다. 그래서 서버 함수가 그냥 죽어버렸고 응답이 없었던 것.

더 근본적인 문제는 POST 핸들러 전체에 try-catch가 없었다는 거다. 뭔가 예외가 터지면 Next.js가 그냥 빈 응답을 돌려줬고, 클라이언트 측에서는 "서버 오류 (빈 응답)"이라는 메시지만 보였다.

해결: 로그 쓰기는 try-catch로 감싸서 실패해도 무시하게 함. POST 핸들러 전체도 outer try-catch 추가.

### 3라운드: JSON 파싱 실패
에러가 바뀌었다. 이번엔 "Claude 응답 파싱 실패"라고 뜨면서 잘린 JSON이 보였다.

속도 개선을 위해 max_tokens를 8192에서 4096으로 줄였는데, 블로그 글은 4096 토큰을 가볍게 넘었다. Claude가 JSON 응답을 중간에 잘라버렸고, 파싱이 실패했던 것. 너무 당연한 원인인데 한참 다른 곳을 뒤졌다.

max_tokens 다시 8192로 복원.

## 결정의 이유
- **Giphy 완전 제거**: 자동 삽입보다 직접 고르는 게 글 품질에도 낫다. 타임아웃 문제도 해결. 일석이조.
- **로그 쓰기 try-catch**: Vercel에서는 어차피 안 되는 기능. 로컬 전용으로 남기되 실패해도 조용히 넘어가게.
- **max_tokens 8192 유지**: 속도보다 완성도가 더 중요하다. 잘린 글이 나오면 의미가 없다.

## 그때 어떤 기분이었나
같은 에러가 계속 나오는데 원인이 매번 달랐다. 하나 고치면 다른 에러가 나오는 식. "이거 고치면 되겠지?" 하고 고쳤는데 또 다른 문제가 튀어나오면 좀 허탈하다. 마치 두더지 게임 같았다.

그래도 세 번째 라운드에서 max_tokens 문제를 발견했을 때는 좀 웃겼다. 너무 당연한 원인인데 왜 그걸 처음부터 안 봤지? 싶어서. 빠르게 만들려다가 오히려 더 오래 걸린 셈이다.

## 배운 것 / Takeaway
- **에러 메시지가 없는 에러가 제일 무섭다.** "서버 오류 (빈 응답)"은 아무 정보도 안 준다. 처음부터 outer try-catch로 모든 에러를 잡아서 응답에 포함시켰어야 했다.
- **환경 차이를 항상 의심하라.** 로컬에서는 잘 돌아가는데 배포하면 안 되는 건 보통 파일시스템, 환경변수, 타임아웃 같은 차이에서 온다.
- **성능 최적화는 동작 확인 후에.** max_tokens 줄여서 속도 올리려다 오히려 버그만 만들었다.
- **핵심 한 줄**: 에러가 없는 게 아니라 에러가 숨어 있던 거였다.
