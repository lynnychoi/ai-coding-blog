# AI 글 수정 기능 — 모델 교체와 원칙 파일 누락 삽질

## 프로젝트 한 줄 소개
기획자/디자이너 출신 Lynn의 개인 블로그. Claude Code로 AI 코딩을 처음 배우는 과정 기록.
글을 쓰거나 수정할 때 AI가 내 블로그 원칙(톤, 문체, 금지어 등)을 참고해서 도와주는 기능이 있다.

## 이 작업을 하게 된 이유
어드민 페이지에서 AI로 글을 쓰거나 수정하는 기능을 만들었는데, 결과물이 영 내 스타일이 아니었다.
처음엔 왜 그런지 몰랐는데 알고 보니 두 가지 문제가 겹쳐 있었다.

첫 번째: API 비용 아끼려고 Groq(Llama)으로 갈아탔는데, 한국어 블로그 글 퀄리티가 Claude에 비해 너무 떨어졌다.
두 번째: 원칙 파일을 만들어뒀는데 (`prompts/LYNN-BLOG-PRINCIPLES.md`), 타입별 확장 파일(`-DEV.md`, `-WRITING.md`)이 API 라우트에서 아예 참조가 안 되고 있었다.

## 뭘 했나 (작업 목록)
- Anthropic 크레딧 충전 후 Groq → Claude (`claude-opus-4-6`) 전환
  - `app/api/write/route.ts`, `app/api/admin/generate-edit/route.ts`, `app/api/edit/[slug]/route.ts`, `lib/ai.ts` 전부 수정
- 타입별 원칙 파일 로딩 추가
  - 마크다운 frontmatter에서 `type: dev` / `type: writing` 파싱
  - dev면 `LYNN-BLOG-PRINCIPLES-DEV.md`, writing이면 `LYNN-BLOG-PRINCIPLES-WRITING.md` 추가 로드
  - 기존 공통 원칙 파일 + 타입별 확장 파일을 합쳐서 system prompt로 넘김

## 막혔던 것들 / 삽질
- **원칙 파일이 있는데 왜 글이 이상하지?** 처음엔 Groq 모델 문제인 줄만 알았다. Claude로 바꿔도 뭔가 아쉬웠는데, 코드 뜯어보니 타입별 확장 파일을 아예 안 읽고 있었다. 공통 파일만 읽고 끝.
  → frontmatter에서 type 파싱해서 조건부로 확장 파일 읽도록 수정
- **Groq 모델 퀄리티 문제**: `llama-3.3-70b-versatile`이 한국어로 짧고 날 것의 블로그 글 쓰는 건 잘 못 한다. 톤 유지가 특히 어렵다.
  → 결국 Claude로 복귀. 비용 아끼려다 퀄리티 포기하는 건 이 블로그 목적상 말이 안 됨.
- **Anthropic 크레딧 충전 방법 모름**: 처음에 어디서 충전하는지 몰랐다. Claude에게 물어봐서 해결.

## 결정의 이유 (왜 이렇게 만들었나)
- 모델: Claude vs Groq 선택에서, 한국어 블로그 특성상 뉘앙스와 톤이 중요하다. 비용보다 품질 우선.
- 원칙 파일: 타입을 분리한 건 dev 글과 writing 글이 구조가 아예 다르기 때문. dev는 삽질 중심 기술 글, writing은 감상/에세이. 하나의 원칙으로 두 개를 다 커버하기 어렵다.

## 그때 어떤 기분이었나
Groq으로 바꾼 건 솔직히 좀 억울했다. 비용 아끼려고 했는데 결과가 별로라서 다시 돌아온 느낌.
근데 원칙 파일 문제를 발견했을 때는 좀 허탈했다. 파일을 만들어놨는데 코드에서 안 읽고 있었던 거잖아.
"내가 원칙을 왜 참고 안 하냐"고 Claude한테 물어봤더니 코드에서 참조 자체가 없었다는 걸 알게 됐다.
고치고 나서 글 품질이 확 달라진 건 아닌데, 적어도 의도한 방향대로 흘러가고 있다는 게 확인이 됐다.

## 배운 것 / Takeaway
- 파일 만들어놓는 것과 코드에서 실제로 읽는 것은 다르다. 만들었다고 자동으로 쓰이는 게 아님.
- 비용 절감 목적으로 모델 바꿀 때는 퀄리티 테스트를 먼저 해봐야 한다. 특히 언어 특성이 중요한 경우.
- AI한테 "왜 이렇게 했냐"고 물어보면 원인 찾는 데 생각보다 빠르다.
- **핵심 한 줄**: 원칙 파일을 만든다고 끝이 아니다. 코드가 그걸 읽고 있는지 확인해야 한다.
