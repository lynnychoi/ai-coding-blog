# scratch-notes 작성 원칙

> 이 파일은 scratch-notes 항목을 작성하는 방법을 정의한다.
> Claude Code가 이 기록만 보고도 블로그 글을 쓸 수 있을 정도의 디테일을 목표로 한다.

---

## 왜 이 기록을 남기는가

- 블로그 글의 **원재료**다. 나중에 `/admin/write` 페이지에 붙여넣어 Claude가 블로그 글로 다듬는다.
- 기록하는 시점에는 기억이 생생하다. 나중에 블로그 쓸 때는 디테일을 잊는다.
- Claude Code는 이 기록만 보고 **프로젝트 맥락, 문제, 감정, 교훈**을 파악해야 한다. 배경 설명 없이 뜬금없이 읽어도 블로그 글이 나와야 한다.

---

## 저자 프로필 (모든 기록에 공통 적용)

- **이름**: 최유경 (Lynn Choi)
- **배경**: 기획자/디자이너 출신. 개발 경험 없음. Claude Code로 AI 코딩 처음 시작.
- **상황**: 서울, 육아 중, 틈새시간에 작업
- **독자**: 개발자가 아닌 기획자·디자이너·AI 코딩 입문자
- **블로그 목적**: Claude Code로 개발하면서 겪은 실수와 교훈을 솔직하게 기록

## 항목 템플릿

각 파일은 아래 섹션을 모두 포함해야 한다. 짧아도 괜찮지만 **빠지면 안 된다**.

```markdown
# [작업 제목]

## 프로젝트 한 줄 소개
이 프로젝트가 뭔지, 누구를 위한 건지, 왜 만들고 있는지.
처음 보는 사람도 맥락을 알 수 있게.

## 이 작업을 하게 된 이유
왜 이걸 만들어야 했나? 어떤 불편함이나 필요에서 출발했나?
"그냥 기능 추가"가 아니라 **왜 지금 이게 필요했는지**.

## 뭘 했나 (작업 목록)
- 만든 것, 바꾼 것, 결정한 것을 구체적으로
- 파일명, 기능명 포함
- 어떤 방향으로 설계했는지 이유도 한 줄씩

## 막혔던 것들 / 삽질
- 뭐가 안 됐는지
- 왜 안 됐는지 (원인)
- 어떻게 해결했는지 (또는 아직 미해결이면 그것도)
- 해결 못 한 채 넘어갔으면 반드시 기록

## 결정의 이유 (왜 이렇게 만들었나)
여러 선택지 중 왜 이 방향을 골랐는지.
나중에 "왜 이렇게 짰지?" 하고 의아할 수 있는 것들.

## 그때 어떤 기분이었나
솔직하게. 막막했는지, 뿌듯했는지, 허탈했는지.
블로그 톤이 이걸로 결정된다. 감정 없이는 글이 재미없어진다.

## 배운 것 / Takeaway
- 기술적으로 새로 알게 된 것
- 태도나 접근 방식에서 달라진 것
- 다음에 똑같은 상황이 오면 어떻게 할 것인지
- **핵심 한 줄**: 이 작업을 한 줄로 압축하면?
```

## 잘 쓴 기록 vs 못 쓴 기록

### ❌ 이렇게 쓰면 블로그 글 못 씀
```
- edit 페이지 만들었음
- giphy 추가함
- 저장 버튼 만들었음
```
→ 왜 했는지, 뭐가 힘들었는지, 뭘 느꼈는지 없음. Claude가 읽어도 글 못 씀.

### ✅ 이렇게 써야 블로그 글 나옴
```
## 이 작업을 하게 된 이유
수정 페이지가 너무 단순했다. 마크다운을 raw로 봐야 해서
어디가 h1인지 h2인지 구분도 안 됐고, 휴대폰에서 쓸 때
특히 헷갈렸다. GIF도 어디 들어가 있는지 눈에 안 보이고.

## 막혔던 것들
- gray-matter가 날짜를 Date 객체로 파싱해버려서
  string으로 넘겨야 하는 API에서 타입 오류 남.
  → instanceof Date 체크 후 .toISOString().substring(0,10) 처리
- 미리보기 띄우려고 react-markdown 쓰려 했는데
  이미 package.json에 있었음. 설치한다고 시간 날릴 뻔.

## 그때 어떤 기분이었나
편집 페이지 다 만들고 나서 뭔가 진짜 쓸 수 있는 도구가
생긴 느낌이었다. 어드민 페이지 들어가서 글 하나 수정해봤는데
실제로 돌아가는 거 보고 좀 신기했음.
```

## 작성 타이밍

- 작업 **직후** 바로 쓰는 게 제일 좋다. 기억이 생생할 때.
- 길게 쓸 필요 없다. 각 섹션 2~5줄이면 충분.
- 완벽하게 쓰려고 미루지 말 것. 짧고 날 것이 낫다.

## Notes를 잘 써야 글이 잘 나온다

Claude는 notes에 있는 것만 쓸 수 있다.
notes가 얕으면 글이 얕다. 이건 원칙 문제가 아니라 **재료 문제**다.

아무리 글쓰기 원칙이 잘 갖춰져 있어도, notes가 목록 세 줄이면 글도 그 수준에서 나온다.

### 좋은 notes에 들어가야 할 것

**과정 (dev 글)**
- 뭘 하려고 했는지 (의도)
- 뭐가 안 됐는지 — "안 됐다" 말고 어떻게 안 됐는지 구체적으로
- 처음에 뭘 시도했다가 실패했는지
- 어떻게 고쳤는지 (단계)
- 해결 못 한 것도 솔직하게

**감정 (dev/writing 공통)**
- 그 순간 뭐가 황당했는지, 뭐가 뿌듯했는지
- 왜 그 감정이었는지 (감정 단어 하나로 끝내지 않기)
- 예상과 결과가 달랐던 순간

**맥락 (writing 글)**
- 어떤 상황에서 이 생각이 시작됐는지
- 구체적인 순간 하나 (추상적인 주제 말고 장면 하나)
- 그때 실제로 한 말, 들은 말

### Notes 셀프 테스트

notes를 다 쓴 후, 이 질문에 답해봐라:

> "이 notes만 보고 Claude가 그 경험을 재현할 수 있나?"

재현 못 할 것 같으면 더 써야 한다.

### 나쁜 notes 예시 (dev)
```
- Playwright 테스트 설정
- write→edit 리다이렉트 확인
- 공통 컴포넌트 테스트
```
→ 이건 할 일 목록이지 경험 기록이 아니다. Claude가 살을 붙일 재료가 없다.

### 좋은 notes 예시 (dev)
```
기능이 하나씩 붙으면서 뭔가 깨질까봐 불안해졌다.
매번 직접 클릭해서 확인하는 게 일이 됐는데, 어느 날 태그 필터가
언제 깨진지도 모르고 배포해버린 걸 나중에 발견함. 그때 좀 충격이었다.
Playwright 도입하면서 PM 시절 QA 체크리스트 만들던 거랑 같은 거라는
생각이 들었다. 근데 사람 대신 로봇이 돌리는 것. 이게 좀 신기했음.
```

### 나쁜 notes 예시 (writing)
```
육아 힘들다
카페가 답인 것 같다
```

### 좋은 notes 예시 (writing)
```
날짜 0318. 아이랑 단둘이 있는 날.
아무도 도와주러 오지 않아도 괜찮다고 생각했지만
막상 혼자 있어 보면 아이가 잠들면 설거지하거나 젖병 세척한다고
시간이 훌쩍 지나간다. 잠을 전혀 보충하지 못하고 하루가 가고.
혼자 있는 날에는 카페에 나가야 할 것 같다.
잠을 못 자는 건 마찬가지지만 뭐라도 나의 일을 하게 되니까.
```
