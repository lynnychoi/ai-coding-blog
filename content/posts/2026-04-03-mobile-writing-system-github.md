# 아이 재우고 누워서 블로그 글 쓰는 시스템을 만들었다

블로그를 만들었다. 글도 몇 개 썼다. 근데 문제가 하나 있었다.

글을 쓰려면 노트북을 열어야 한다. 마크다운 파일을 만들고, 내용을 쓰고, 커밋하고, 푸시해야 한다.

> **커밋이란?** 파일의 변경사항을 저장소에 기록하는 거다. 일종의 "저장" 버튼인데, 언제 뭘 바꿨는지 이력이 남는다.

이게 앉아서 작업할 수 있는 상황이면 괜찮다. 근데 나는 육아 중이다. 노트북 앞에 앉을 시간이 *그렇게* 많지 않다. 아이 재우고 나면 나도 같이 누워 있게 되는데, 그 시간에 머릿속에서 글감이 떠오른다. 그걸 메모장에 적어두고, 나중에 노트북 켜서 다시 옮기고… 이 과정이 점점 귀찮아졌다.

그래서 생각했다. **휴대폰으로 메모 몇 줄 던지면 알아서 블로그 글이 되는 구조**를 만들면 안 되나?

---

## 구조를 설계하다 — DB 없이 GitHub으로

이 블로그는 글을 데이터베이스에 저장하지 않는다. 마크다운 파일을 GitHub 저장소에 올려두면, 그걸 읽어서 보여주는 방식이다.

> **GitHub이란?** 코드나 파일을 저장하고 관리하는 온라인 저장소다. 구글 드라이브 같은 건데, 변경 이력이 다 남는다는 점이 다르다.

그래서 글쓰기 시스템도 같은 방식으로 가기로 했다. 휴대폰에서 메모를 쓰면, 그 내용이 Claude에게 전달되고, Claude가 블로그 글 형식으로 다듬어서 GitHub에 직접 올린다. GitHub에 파일이 올라가면 Vercel이 자동으로 감지해서 사이트를 다시 배포한다.

> **Vercel이란?** 웹사이트를 인터넷에 올려주는 서비스다. GitHub이랑 연결해두면 파일이 바뀔 때마다 알아서 최신 버전으로 업데이트해준다.

별도 데이터베이스를 만들 필요가 없다. GitHub이 저장소 역할을 하고, Vercel이 배포를 맡고, Claude가 글을 다듬는다. 각자 자기 역할만 하면 된다.

---

## 만든 것들

실제로 만든 파일들을 정리하면 이렇다.

**글쓰기 룰 문서** — Claude가 글을 쓸 때 지켜야 할 원칙을 정리한 파일이다. 톤, 포맷, GIF 삽입 규칙, 기술 용어 설명 방식 같은 것들. 독자가 개발자가 아니니까 이 룰이 사실 제일 중요했다. Claude한테 "알아서 써 줘"라고 하면 개발 블로그처럼 써버린다.

**GitHub 연결 코드** — GitHub에 파일을 읽고 쓸 수 있는 함수 두 개를 만들었다. 하나는 파일 읽기, 하나는 파일 올리기(커밋). 이게 있어야 휴대폰에서 쓴 글이 GitHub까지 도달한다.

> **API란?** 두 프로그램이 서로 대화하는 창구다. 여기서는 내 블로그 코드가 GitHub한테 "이 파일 좀 올려 줘"라고 요청하는 통로라고 보면 된다.

**인증 시스템** — 아무나 글을 올릴 수 있으면 안 되니까. 어드민 페이지는 로그인한 사람만 쓸 수 있게 하고, 공개 페이지는 비밀번호를 입력해야 쓸 수 있게 만들었다.

**글쓰기 페이지** — 메모를 입력하고, 이미지를 첨부할 수 있는 페이지. 제출하면 Claude가 글을 만들고, 결과가 GitHub에 바로 올라간다.

## 삽질 기록 — 경로 하나 때문에 한 시간

제일 시간을 많이 잡아먹은 건 허무하게도 **파일 경로 문제**였다.

코드에서 다른 파일을 불러올 때 경로를 적는다. `@/lib/github` 이런 식으로 쓰면 프로젝트 최상위부터 찾아가는 건데, 이게 동작하려면 설정 파일에 "@는 여기를 가리킨다"는 걸 적어둬야 한다. ~~당연히 되어 있을 줄 알았다.~~ 안 되어 있었다.

그래서 `@/lib/github`이라고 쓸 때마다 "그런 파일 없는데?" 에러가 났다. 처음엔 파일 이름을 잘못 쓴 줄 알고, 오타를 찾고, 파일을 옮겨보고… 한참 삽질하다가 설정 문제라는 걸 깨달았다. 결국 모든 파일에서 `@/lib/...`을 `../../../lib/...` 같은 상대 경로로 바꿨다.

*이런 걸 한 파일도 아니고 여러 파일에서 하나하나 바꾸는 거다.* 기능적으로는 아무것도 바뀌지 않는데 시간만 갔다.

---

## 이미지가 깨지는 이유

글에 이미지도 넣을 수 있게 만들고 싶었다. 휴대폰으로 찍은 사진을 첨부하면 GitHub에 같이 올라가는 식으로.

근데 이미지는 텍스트 파일이 아니다. 마크다운 파일은 그냥 글자 그대로 올리면 되지만, 이미지는 *바이너리 데이터*라서 다른 방식으로 처리해야 한다.

> **바이너리란?** 컴퓨터가 이해하는 0과 1로 된 데이터다. 텍스트는 사람이 읽을 수 있지만, 이미지나 동영상 같은 건 0과 1 덩어리로 저장된다.

처음에 텍스트 파일처럼 그냥 올렸더니 이미지가 깨졌다. 알아보니 GitHub API에 파일을 올릴 때는 *base64*라는 형식으로 변환해야 한다.

> **base64란?** 바이너리 데이터를 텍스트 형태로 바꿔주는 인코딩 방식이다. 이미지 같은 파일을 "글자"로 표현해서 전송할 수 있게 해준다.

변환 과정이 이렇다:

```
파일(File) → 바이트 배열(arrayBuffer) → base64 텍스트 → GitHub API로 전송
```

이 흐름을 파악하는 데 시간이 걸렸다. 근데 한 번 이해하고 나니까, 이미지든 PDF든 뭐든 같은 방식으로 올릴 수 있다는 게 보였다.

## 인증을 두 가지로 만든 이유

글쓰기 페이지를 두 버전으로 만들었다. 하나는 어드민 전용, 하나는 공개용.

어드민 페이지는 이미 로그인한 상태에서 쓰는 거라 별도 비밀번호 입력이 필요 없다. 브라우저에 저장된 쿠키로 "이 사람 어드민 맞아"를 확인한다.

> **쿠키란?** 웹사이트가 브라우저에 남겨두는 작은 메모다. "이 사람 로그인했음"이라고 적어두면, 다음에 페이지를 열 때 다시 로그인 안 해도 된다.

공개 페이지는 URL만 알면 접근할 수 있지만, 글을 쓰려면 비밀번호를 입력해야 한다. 이건 내가 휴대폰에서 빠르게 접근하고 싶을 때를 위한 거다. 어드민 로그인 과정을 거치지 않고, 비밀번호 한 번만 치면 바로 글을 쓸 수 있다.

하나의 API에서 두 방식을 모두 처리하게 만들었다. "쿠키가 있으면 통과, 없으면 비밀번호 확인" — 이 로직이 한 곳에 있는 거다.

## Edge 런타임이라는 복병

인증 코드를 만들면서 예상 못한 문제를 만났다. 암호화 방식을 고르는 건데, 이게 생각보다 까다로웠다.

> **런타임이란?** 코드가 실제로 돌아가는 환경이다. 같은 코드라도 어디서 돌리느냐에 따라 쓸 수 있는 기능이 다르다.

Next.js에는 두 가지 런타임이 있다. 하나는 *Node 런타임*이라고 해서 서버에서 돌아가는 것, 하나는 *Edge 런타임*이라고 해서 사용자와 가까운 곳에서 빠르게 돌아가는 것. 내 블로그에서는 인증 체크를 Edge에서 하고, 실제 글쓰기 처리는 Node에서 한다.

문제는 **두 런타임에서 쓸 수 있는 암호화 도구가 다르다**는 거다. Node에서만 되는 게 있고, Edge에서만 되는 게 있다. 둘 다에서 동작하는 걸 찾아야 했다.

결국 `crypto.subtle`이라는 웹 표준 암호화 방식을 선택했다. 이건 브라우저에서도 되고, Node에서도 되고, Edge에서도 된다. 범용적인 대신 코드가 좀 더 길어지긴 했는데, 런타임 때문에 나중에 다시 바꾸는 것보다 낫다고 판단했다.

## 결과

이제 이런 흐름이 가능해졌다.

1. 아이 재우고 침대에 눕는다
2. 휴대폰으로 글쓰기 페이지 접속
3. 메모 몇 줄 + 이미지 첨부
4. 제출하면 Claude가 블로그 글 형식으로 다듬어줌
5. 결과가 GitHub에 바로 올라감
6. Vercel이 자동으로 블로그에 반영

별도 데이터베이스 없이, 복잡한 서버 없이, GitHub 하나로 저장과 배포가 다 된다. 기획자 시절에 *그렇게나* 싫어했던 "파일 전달" 방식이 여기서는 오히려 가장 심플한 정답이었다.

> 복잡한 인프라 없이 GitHub을 저장소처럼 쓰는 구조가, 생각보다 잘 작동한다. 거창한 시스템보다 내 상황에 맞는 구조가 더 중요했다.
