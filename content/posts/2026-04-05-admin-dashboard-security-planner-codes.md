# 기획자가 보안을 만든다는 게 말이 돼?

글이 쌓이기 시작하면 관리가 필요해진다. 어떤 글이 공개 상태이고, 어떤 글이 아직 작업 중인지. 수정하고 싶을 때 바로 들어갈 수 있는 곳. 기획자 시절에 *그렇게나* 만들었던 어드민 페이지를, 이번엔 내 블로그에 직접 만들어야 했다.

근데 어드민 페이지는 나만 들어갈 수 있어야 한다. 내 블로그인데 아무나 글을 올리거나 수정할 수 있으면 곤란하니까. 그래서 이 작업은 사실 두 가지였다. **대시보드를 만드는 것**, 그리고 **그걸 잠그는 것**.

![Security Breach Reaction GIF](https://media.giphy.com/media/tLEwRu07PWnePmfrVR/giphy.gif)

## 대시보드부터 — 글 목록을 한눈에

만든 건 `/admin` 경로의 대시보드 페이지다. dev 글, writing 글을 타입별로 나눠서 보여주고, 각 글마다 "보기"와 "수정" 링크가 붙는다.

> **대시보드란?** 관리자가 전체 현황을 한눈에 볼 수 있는 화면이다. 비행기 조종석 계기판 같은 거라고 생각하면 된다.

기획자 시절엔 이런 페이지의 *와이어프레임*을 그리는 쪽이었다. "여기에 글 목록이 나오고, 이 버튼을 누르면 수정 페이지로 갑니다" 이런 걸 문서로 정리해서 개발자에게 전달했다. 근데 지금은 그 문서를 Claude에게 전달하고 있다. 역할은 같은데 상대가 바뀐 셈.

글 목록을 가져오는 건 별도 API를 만들었다.

> **API란?** 두 프로그램이 서로 대화하는 창구다. 대시보드 화면이 "글 목록 줘" 하고 요청하면, API가 마크다운 파일들을 읽어서 목록을 돌려주는 구조.

여기서 한 가지 결정을 해야 했다. 대시보드 페이지를 *서버 컴포넌트*로 만들 건지, *클라이언트 컴포넌트*로 만들 건지.

## 서버 컴포넌트? 클라이언트 컴포넌트?

Next.js에는 두 종류의 컴포넌트가 있다.

> **서버 컴포넌트**는 서버에서 미리 데이터를 준비해서 완성된 화면을 보내준다. 식당에서 완성된 요리가 나오는 것과 비슷하다.
>
> **클라이언트 컴포넌트**는 브라우저에서 실행된다. 버튼 클릭, 입력창 타이핑 같은 사용자 상호작용을 처리할 수 있다. 뷔페에서 직접 접시에 담는 느낌이랄까.

대시보드는 글 목록을 서버에서 바로 읽어오는 게 좋다. 클라이언트에서 데이터를 가져오면 페이지가 열릴 때 잠깐 빈 화면이 깜빡이거든. 그래서 서버 컴포넌트로 만들었다.

근데 로그아웃 버튼이 문제였다. 버튼을 누르면 쿠키를 지우고 로그인 페이지로 보내야 하는데, 이 "누르면"이라는 동작이 클라이언트에서만 가능하다. 서버 컴포넌트에서는 `onClick` 같은 걸 쓸 수 없다.

~~처음엔 그냥 페이지 전체를 클라이언트 컴포넌트로 바꾸면 되지~~ 싶었다. 근데 그러면 서버사이드 데이터 패칭의 장점을 다 잃는다. 깜빡임이 다시 생긴다는 거다.

결국 **로그아웃 버튼만 별도 클라이언트 컴포넌트로 분리**했다. `LogoutButton.tsx`라는 파일을 따로 만들어서, 대시보드 페이지(서버 컴포넌트) 안에 끼워 넣는 방식이다. 페이지 전체는 서버에서 돌아가고, 버튼 하나만 브라우저에서 동작한다.

이게 Next.js의 기본 패턴이라고 Claude가 알려줬다. *필요한 부분만 클라이언트로 분리한다.* 처음엔 왜 이렇게 복잡하게 하나 싶었는데, 이유를 알고 나니 납득이 됐다.

## 잠금장치 — middleware로 문 잠그기

대시보드를 만들었으면 이제 잠가야 한다. `/admin`으로 들어오는 모든 요청을 검사해서, 로그인 안 된 사람은 로그인 페이지로 돌려보내야 한다.

> **middleware란?** 요청이 페이지에 도달하기 전에 가로채서 검사하는 관문이다. 건물 입구의 보안 게이트 같은 거. 출입증(쿠키) 없으면 안으로 못 들어간다.

`middleware.ts`라는 파일을 프로젝트 루트에 만들면 된다. 이 파일이 `/admin`으로 시작하는 모든 경로를 감시한다. 쿠키에 인증 토큰이 없으면? 로그인 페이지로 *redirect*.

> **redirect란?** "여기 말고 저쪽으로 가세요" 하고 자동으로 페이지를 바꿔주는 것이다.

여기서 처음 보는 개념이 나왔다. *Edge 런타임*이니 *Node 런타임*이니 하는 것들.

> **런타임이란?** 코드가 실행되는 환경이다. 같은 코드라도 어디서 돌리느냐에 따라 쓸 수 있는 기능이 다르다. Edge 런타임은 가볍고 빠르지만 할 수 있는 게 제한적이고, Node 런타임은 무겁지만 할 수 있는 게 많다.

middleware는 Edge 런타임에서 돌아간다. 그래서 Node.js에서만 쓸 수 있는 기능을 middleware 안에서 쓰면 에러가 난다. Claude가 이걸 미리 알려줘서 삽질을 피했는데, 설명을 듣는 것과 *왜 이렇게 나눠져야 하는지* 체감하는 건 다른 문제였다. 솔직히 아직 100% 와닿지는 않는다.

그리고 한 가지 찝찝한 게 있다. Next.js 16에서는 `middleware.ts`라는 이름이 곧 없어지고 `proxy.ts`로 바꿔야 한다는 경고가 뜬다. 빌드는 되는데 터미널에 경고가 남아 있다. ~~내일의 나에게 미루는 중이다.~~

## API 키를 숨기는 법 — Giphy 프록시

블로그 글에 GIF를 넣기 위해 Giphy API를 쓰고 있다. 근데 이걸 브라우저에서 직접 호출하면 문제가 생긴다. 브라우저의 개발자도구를 열면 요청 내역이 다 보이는데, 거기에 **API 키가 그대로 노출**된다.

> **API 키란?** 특정 서비스를 이용하기 위한 비밀번호 같은 것이다. 이게 노출되면 다른 사람이 내 이름으로 API를 마구 호출할 수 있다.

그래서 *프록시*를 만들었다.

> **프록시란?** 대리인이다. 브라우저가 직접 Giphy에 요청하는 대신, 내 서버가 중간에서 대신 요청해주는 구조. 브라우저 → 내 서버 → Giphy. API 키는 내 서버에만 있으니까 브라우저에서는 절대 보이지 않는다.

`app/api/admin/giphy/route.ts`라는 파일이 이 역할을 한다. 브라우저에서 "'cat' GIF 검색해줘" 하고 내 서버에 요청하면, 서버가 Giphy API 키를 붙여서 Giphy에 대신 요청하고, 결과만 브라우저에 돌려준다.

솔직히 처음엔 *이걸 꼭 이렇게까지 해야 하나* 싶었다. 귀찮으니까. 근데 한번 개발자도구에서 API 키가 그대로 찍힌 걸 보고 나니 생각이 바뀌었다. 기획자 시절에 보안 요구사항 문서 쓸 때는 "API 키 노출 방지" 한 줄이면 끝이었는데, 직접 구현해보니 그 한 줄 뒤에 이런 구조가 있었구나 싶었다.

## dev server가 두 번 뜨면 생기는 일

작업 중에 사소한 삽질도 하나 했다. 개발 서버(`next dev`)가 이미 돌아가고 있는데 터미널을 새로 열어서 또 실행했더니, `.next/dev/lock`이라는 파일이 충돌하면서 서버가 안 뜨는 거다.

잠금 파일이 이미 있으니 "누군가 이미 쓰고 있다"고 판단하는 것이었다. 해결은 간단했다.

```bash
pkill -f "next dev"
```

돌아가고 있는 모든 `next dev` 프로세스를 죽이고 다시 시작하면 된다. 별거 아닌데, 처음 만나면 *뭐가 잘못된 거지* 하고 당황한다.

## 어드민을 만들면서 느낀 것

기획자 시절에 어드민 페이지는 늘 만들어야 하는 것 중 하나였다. 근데 항상 "이런 기능이 필요합니다"를 정리하는 쪽이었지, 그걸 직접 잠그고 보호하는 건 내 영역이 아니었다.

이번에 직접 해보니까, 보안이라는 게 거창한 게 아니었다. 쿠키를 확인하고, 없으면 돌려보내고, API 키를 서버에만 두고. 개념 자체는 단순하다. 다만 **그 개념들이 코드로 어떻게 연결되는지**를 처음 보는 거라 시간이 걸렸을 뿐이다.

Claude가 코드를 써주더라도, 왜 이 코드가 필요한지는 내가 이해해야 한다. middleware가 왜 Edge에서 도는지, 왜 클라이언트 컴포넌트를 분리해야 하는지, 왜 프록시가 필요한지. 그걸 이해하고 나면 코드 자체는 Claude에게 맡겨도 된다.

> 보안의 코드를 쓰는 건 Claude가 해도 된다. 근데 **뭘 왜 잠가야 하는지**는 내가 알아야 한다.
