  # 종합 편집 페이지 구축

  ## 프로젝트 한 줄 소개
  Claude Code로 만드는 개인 블로그. 기획자 출신 Lynn이 AI 코딩 과정을 기록. 글 작성·수정을 휴대폰에서도 쉽게 하는 게 목표.

  ## 이 작업을 하게 된 이유
  기존 수정 페이지가 너무 단순했다. 텍스트 하나 바꾸려면 마크다운 raw 텍스트를 눈으로 훑어야 했는데, `---` 로 시작하는 frontmatter, `#` 헤더, `![gif](url)` 이런 것들이 뒤섞여 있어서 실제 내용이 어디에 있는지 파악하기 어려웠다. 특히 휴대폰에서는 더 심했다. 수정하고 싶은 게 생겼을 때 "이거 쓸 수 있는 도구가 아니다"라는 느낌이 들었다.

  ## 뭘 했나
  - **`app/api/admin/posts/[slug]/route.ts`** — GET: 글 가져오면서 `gray-matter`로 frontmatter 파싱해서 필드별로 분리해서 반환. PUT: 마크다운 직접 저장 (Claude 없이).
  - **`app/api/admin/generate-edit/route.ts`** — Claude AI 수정 API. 커밋은 안 하고 수정된 마크다운만 반환. 사용자가 확인 후 저장하는 구조.
  - **편집 페이지 전면 재작성**:
    - 제목, 날짜, 타입, 태그를 각각 편집할 수 있는 필드
    - 원본 메모/소스 섹션 (접기/펼치기, 수정 가능) — 나중에 이게 블로그 재료가 됨
    - 마크다운 편집기 (편집 ↔ 미리보기 토글)
    - 하단 탭: [AI 수정] [GIF] [이미지]
    - AI 탭: 지시 입력 → Claude 수정안 → 확인 후 저장
    - GIF 탭: Giphy 검색 → 클릭하면 커서 위치에 삽입
    - 이미지 탭: 업로드 → GitHub 커밋 → 커서 위치에 마크다운 삽입
    - 상단 고정 저장 버튼 (변경사항 있을 때만 활성화)

  ## 막혔던 것들 / 삽질
  - **frontmatter의 날짜가 Date 객체로 파싱됨**: `gray-matter`가 YAML의 날짜를 자동으로 `Date` 객체로 변환한다. API에서 string으로 반환해야 해서 `instanceof Date ? .toISOString().substring(0,10) : String(data.date)` 처리를 넣어야 했다.
  - **body 분리 로직**: 전체 마크다운에서 frontmatter 제거하고 첫 번째 `# 제목` 줄도 제거해서 순수 본문만 남겨야 했다. 정규식 패턴 몇 번 틀렸다.
  - **AI 수정 후 상태 동기화**: Claude가 수정한 전체 마크다운을 받았을 때, 다시 frontmatter/title/body로 분리해서 각 state에 반영해야 했다. 처음엔 전체 마크다운을 하나의 state로 관리했다가, 나중에 필드 분리 구조로 바꾸면서 복잡해졌다.
  - **커서 위치 삽입**: GIF나 이미지를 textarea의 커서 위치에 끼워 넣는 게 생각보다 까다로웠다. `selectionStart`/`selectionEnd` + `requestAnimationFrame`으로 처리.

  ## 결정의 이유
  - AI 수정이 즉시 커밋되지 않고 미리보기 후 저장하는 구조: 실수로 글이 망가지는 걸 막기 위해. "Claude가 고쳤는데 더 이상해졌을 때" 되돌릴 수 있어야 한다.
  - frontmatter와 body를 분리해서 state 관리: 제목·날짜 같은 메타 필드를 별도 인풋으로 편집 가능하게 하려면 어쩔 수 없었다. 저장할 때 `buildMarkdown(fields, body)`로 다시 합친다.
  - notes 필드를 frontmatter에 저장: 별도 DB 없이 마크다운 파일 안에 원본 메모를 보존하기 위해.

  ## 그때 어떤 기분이었나
  이 페이지 만드는 데 제일 오래 걸렸다. 기능이 많다 보니 설계를 몇 번 바꿨다. 완성하고 나서 실제로 글 하나 열어봤는데 제목이 큰 인풋으로 뜨고, 본문이 textarea에 있고, GIF 탭에서 검색해서 클릭하니까 바로 삽입되는 거 보고 — 이게 내가 원하던 거라는 느낌이 들었다. 뭔가 진짜 도구가 생긴 것 같은 기분.

  ## 배운 것 / Takeaway
  - 상태(state) 설계를 먼저 하고 코딩하는 게 맞다. 나중에 구조 바꾸면 두 배로 힘들어진다
  - "AI가 수정한다"와 "내가 저장한다"를 분리하는 UX가 중요하다. 자동화를 너무 믿으면 실수가 생긴다
  - gray-matter 날짜 파싱 주의 — 이거 모르면 타입 오류로 한참 헤맨다
  - **핵심 한 줄**: 도구를 만든다는 건 내가 뭘 불편해하는지를 정확히 알아야 한다는 거다.

# 수정 페이지를 진짜 도구로 만든 날

블로그를 직접 만드는 이유 중 하나가 "어디서든 글을 쓰고 수정할 수 있는 환경"이었다. 근데 수정 페이지를 열면 이런 게 나왔다.

```
---
title: "히어로 섹션 하나 만드는 데 하루 다 썼다"
date: 2026-03-15
tags: ["dev", "claude-code"]
---

# 히어로 섹션 하나 만드는 데 하루 다 썼다

처음엔 간단할 줄 알았다...
```

이 덩어리에서 본문을 찾아 수정하고, 저장하는 구조였다. 데스크탑에서도 불편한데 휴대폰으로 열면 더 심했다. "이거 쓸 수 있는 도구가 아니다"라는 생각이 들면 사람은 안 쓰게 된다. 그래서 다시 만들었다.

## 뭘 만들었나

편집 페이지를 전면 재작성했다. 구체적으로는:

- 제목, 날짜, 타입, 태그를 **각각 편집할 수 있는 필드**로 분리
- 마크다운 편집기 — 편집 ↔ 미리보기 토글
- 하단 탭 세 개: [AI 수정] [GIF] [이미지]
- 상단 고정 저장 버튼 (변경사항이 있을 때만 활성화)

그리고 원본 메모 섹션도 따로 뒀다. 블로그 글 쓰기 전에 적어두는 재료 메모인데, 이게 나중에 Claude한테 넘기는 소스가 된다. 접기/펼치기로 평소엔 숨겨두고 필요할 때 꺼내 쓰는 구조.

## 제일 까다로웠던 것들

### 마크다운 파일을 필드로 쪼개기

글은 파일 하나로 저장돼 있다. 그 안에 제목·날짜 같은 메타 정보(*frontmatter*라고 부른다 — 파일 맨 위 `---`로 감싸진 영역)와 실제 본문이 섞여 있다.

> **frontmatter란?** 마크다운 파일 맨 위에 붙는 메타 정보 블록. 제목, 날짜, 태그 같은 것들을 저장한다. `---`로 시작하고 `---`로 끝난다.

편집 페이지에서는 이걸 분리해야 했다. 제목은 큰 인풋으로, 날짜는 날짜 선택기로, 본문은 textarea로 — 각각 따로 편집할 수 있으려면 애초에 쪼개서 관리해야 한다.

`gray-matter`라는 라이브러리로 파싱했는데, 여기서 함정이 하나 있었다. YAML 날짜 형식(`2026-03-15`)을 이 라이브러리가 자동으로 *JavaScript Date 객체*로 변환해버린다. 문자열이 아니라 날짜 타입으로 바뀌는 거다.

> **Date 객체란?** 프로그래밍에서 날짜/시간을 다루는 데이터 타입. `"2026-03-15"`라는 문자열과 다르다. 그냥 출력하면 `Fri Mar 15 2026 09:00:00 GMT+0900...` 이런 식으로 나온다.

~~`2026-03-15`가 왜 저렇게 나오지?~~ 하고 한참 헤맸다. 결국 API에서 반환할 때 `instanceof Date ? 앞 10자리만 잘라내기 : 그냥 문자열로` 처리를 넣어서 해결했다.

### AI가 수정하고, 내가 저장한다

AI 수정 탭에서는 지시를 입력하면 Claude가 본문을 수정해서 보여준다. 근데 이게 **즉시 저장되지 않는다**. 수정안이 미리보기로 뜨고, 내가 확인한 다음에 저장 버튼을 눌러야 실제로 반영된다.

이건 의도한 구조다. "Claude가 고쳤는데 더 이상해졌을 때" 되돌릴 수 있어야 한다. AI를 너무 믿으면 실수가 생긴다 — 이건 AI 코딩을 해보면서 배운 거다.

근데 이 구조 때문에 코드가 복잡해졌다. Claude가 수정한 전체 마크다운을 받으면, 다시 frontmatter/제목/본문으로 쪼개서 각 필드에 반영해야 한다. 처음엔 전체 마크다운을 state 하나로 관리했다가 나중에 필드 분리 구조로 바꾸면서 연쇄적으로 손볼 게 생겼다.

> **state란?** 화면에 표시되는 데이터를 저장하는 공간. 이걸 바꾸면 화면이 자동으로 업데이트된다.

설계를 먼저 잡고 시작해야 하는 이유가 여기 있다. 나중에 구조를 바꾸면 두 배로 힘들어진다.

### 커서 위치에 끼워 넣기

GIF 탭에서 Giphy를 검색하고 클릭하면 본문 textarea의 **커서가 있는 위치**에 GIF 마크다운이 삽입된다. 이미지 업로드도 같은 방식이다.

*커서 위치 삽입*이 생각보다 까다로웠다. textarea에서 현재 커서가 어디에 있는지 알아내고(`selectionStart`), 그 위치에 텍스트를 끼워 넣고, 삽입 후 커서를 올바른 위치로 다시 옮기는 과정이 필요하다. `requestAnimationFrame`이라는 걸 써서 처리했는데 — 이건 "화면이 업데이트된 다음 타이밍에 실행해"라는 뜻이다. 이걸 안 쓰면 커서가 엉뚱한 데로 튄다.

## 완성하고 나서

다 만들고 나서 실제로 글 하나를 열어봤다. 제목이 큰 인풋으로 뜨고, 본문이 textarea에 있고, GIF 탭에서 검색해서 클릭하니까 바로 삽입됐다.

이게 내가 원하던 거였다. 뭔가 **진짜 도구**가 생긴 것 같은 기분.
