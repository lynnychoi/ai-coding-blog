# 누워서 블로그 글 쓰는 시스템을 만들었다

블로그 글을 하나 쓰려면 이 과정을 거쳐야 했다. 컴퓨터를 켠다. VS Code를 연다. 마크다운 파일을 만든다. 글을 쓴다. 터미널에서 커밋한다. 푸시한다.

~~이게 뭐 대수냐~~ 싶겠지만, 아이 재운 직후에 이 루틴을 밟을 에너지는 없다. 침대에 누워서 *아 이거 블로그에 써야 하는데* 싶은 순간이 오면, 그냥 내일로 미룬다. 그러다 까먹는다.

그래서 생각했다. 휴대폰에서 메모 몇 줄만 던지면 글이 만들어지는 구조를 만들 수 없을까?

![Tired Phone GIF by SLOTHILDA](https://media.giphy.com/media/3o6fJ1jAdPFYK4lamI/giphy.gif)

## 구조부터 그렸다

원하는 흐름은 간단했다.

1. 휴대폰에서 메모를 쓴다 (제목, 태그, 본문 재료)
2. 보내면 Claude가 내 글쓰기 스타일로 블로그 글을 만든다
3. 완성된 글이 자동으로 블로그에 올라간다

이걸 위해 필요한 게 세 가지였다.

- **글쓰기 페이지** — 휴대폰 브라우저에서 열리는 폼
- **Claude 연동** — 메모를 받아서 글로 바꿔주는 부분
- **자동 배포** — 완성된 글이 바로 블로그에 반영되는 구조

## 데이터베이스 없이 GitHub을 저장소로 쓰기

보통 글을 저장하려면 데이터베이스가 필요하다.

> **데이터베이스란?** 데이터를 체계적으로 저장하고 꺼내 쓸 수 있는 창고 같은 것. 보통 서버에 따로 세팅해야 한다.

근데 이 블로그는 이미 GitHub에 마크다운 파일로 글을 관리하고 있었다. GitHub에 파일을 올리면 Vercel이 자동으로 감지해서 사이트를 다시 배포한다.

> **Vercel이란?** 웹사이트를 인터넷에 올려주는 서비스. GitHub이랑 연결해두면 코드가 바뀔 때마다 알아서 최신 버전으로 업데이트해준다.

그러면 별도 데이터베이스 없이, **GitHub 자체를 저장소로 쓰면 되는 거 아닌가?** 글쓰기 페이지에서 완성된 마크다운을 GitHub에 직접 올리면, Vercel이 알아서 배포한다. 한 다리 건너뛸 수 있는 구조다.

GitHub은 파일을 올리고 읽을 수 있는 통로를 제공한다.

> **API란?** 두 프로그램이 서로 대화하는 창구. 여기서는 내 블로그 코드가 GitHub한테 "이 파일 저장해줘"라고 요청을 보내는 통로다.

이 API를 감싸는 코드 두 개를 만들었다. 하나는 파일을 읽는 함수, 하나는 파일을 올리는 함수. 텍스트든 이미지든 *base64*라는 형식으로 변환하면 다 올릴 수 있다.

> **base64란?** 파일(이미지, 문서 등)을 텍스트 형태의 긴 문자열로 바꾸는 방식. 이렇게 바꾸면 텍스트만 주고받을 수 있는 통로로도 이미지를 보낼 수 있다.

## Claude한테 내 스타일을 가르쳤다

메모를 던지면 Claude가 글을 쓴다. 근데 아무렇게나 쓰면 안 된다. 내 톤으로, 내 블로그 포맷으로 나와야 한다.

그래서 **글쓰기 룰 문서**를 만들었다. Claude가 글을 쓸 때 이 문서를 먼저 읽고 시작하는 구조다.

룰 문서에 넣은 것들:
- 톤: 솔직하고 털털하게. 설교 금지. 과장 금지
- 포맷: GIF 필수, 기술 용어엔 설명 붙이기, 짧은 문장 선호
- 독자: 개발자가 아닌 사람이 읽는다는 전제
- 구조: 왜 이걸 했는지 → 뭐가 막혔는지 → 뭘 배웠는지

이게 이 시스템에서 **제일 중요한 부분**이다. 자동화가 아무리 잘 돼도 글이 내 목소리가 아니면 의미가 없으니까.

## 보안 — 나만 쓸 수 있어야 한다

글쓰기 페이지가 인터넷에 공개되어 있으면 아무나 글을 올릴 수 있다. 그러면 안 된다.

인증 방식을 두 가지로 나눴다.

| 페이지 | 누가 쓰나 | 인증 방식 |
|--------|-----------|----------|
| `/write` (공개 경로) | 나 (모바일에서) | 비밀번호 입력 |
| `/admin/write` (어드민) | 나 (로그인된 상태) | 쿠키 자동 확인 |

어드민 페이지는 로그인하면 브라우저에 쿠키가 저장되고, 그 쿠키가 있으면 별도 비밀번호 없이 바로 글을 쓸 수 있다. 공개 경로는 쿠키가 없으니까 매번 비밀번호를 입력해야 한다.

> **쿠키란?** 웹사이트가 브라우저에 남기는 작은 메모. "이 사람은 아까 로그인했음"이라는 정보를 담아두는 거다.

근데 여기서 좀 복잡한 문제가 있었다. 이 블로그는 두 가지 환경에서 코드가 돌아간다.

> **Edge 런타임과 Node 런타임**: 웹사이트의 코드가 실행되는 두 가지 환경이다. Edge는 사용자에게 가까운 서버에서 빠르게 실행되는 것, Node는 메인 서버에서 실행되는 것. 문제는 이 둘이 지원하는 기능이 미묘하게 다르다는 거다.

로그인 여부를 확인하는 코드(*middleware*)는 Edge에서 돌고, 실제 글을 처리하는 코드(*API*)는 Node에서 돈다. 암호화 방식이 한쪽에서만 되면 안 된다. 그래서 양쪽 다 지원하는 `crypto.subtle`이라는 방식을 선택했다. 이런 건 직접 부딪혀보기 전엔 모른다.

## 삽질 기록

### 경로 문제로 시간 날린 이야기

코드를 짜다 보면 다른 파일에 있는 함수를 가져다 쓸 때가 많다. 이때 그 파일이 어디 있는지 경로를 적어줘야 한다.

보통은 `@/lib/auth`처럼 깔끔하게 쓸 수 있는데, 그게 되려면 프로젝트 설정 파일에 "@ 이 기호는 프로젝트 최상위 폴더를 뜻한다"고 미리 알려줘야 한다. ~~당연히 설정되어 있는 줄 알았다.~~ 안 되어 있었다.

오류가 났고, 여러 파일에서 동시에 터졌고, 하나씩 고치느라 시간을 꽤 날렸다. 결국 `../../../lib/auth` 같은 상대경로로 전부 바꿨다. 파일 위치에 따라 `../`이 몇 개 붙는지 세야 하는 그 귀찮은 방식.

교훈: **프로젝트 시작할 때 경로 설정부터 확인하자.**

### 이미지가 깨진 이유

글에 이미지를 첨부할 수 있게 만들었다. 이미지 파일을 받아서 GitHub에 올리면 되니까 간단할 줄 알았다.

근데 이미지는 텍스트가 아니다. 텍스트 파일 다루듯이 처리하면 데이터가 깨진다. 이미지 파일을 먼저 바이너리 데이터로 읽고, 그걸 다시 base64 텍스트로 변환한 다음, GitHub API로 보내야 한다.

```
File → arrayBuffer() → Buffer.from() → base64 문자열 → GitHub API
```

이 흐름을 파악하는 데 시간이 걸렸다. 처음엔 왜 이미지가 깨지는지 이유도 몰랐다. *바이너리와 텍스트가 다르다*는 걸 몸으로 배운 순간이었다.

## 실제로 돌아가는 걸 보니

만들고 나서 휴대폰으로 테스트해봤다. 브라우저에서 주소 치고, 메모 적고, 보내기 누르니까 GitHub에 마크다운 파일이 올라가고, 잠시 뒤에 블로그에 글이 뜬다.

*이게 된다고?* 싶었다. 물론 삽질 시간을 합치면 전혀 효율적이지 않았다. 근데 **한 번 만들어놓으면 계속 쓸 수 있는 구조**라는 게 다르다. 매번 컴퓨터 켜고 커밋하는 과정이 사라졌다. 그 귀찮음이 사라지면 글을 더 자주 쓰게 될 거라고 기대하고 있다.
