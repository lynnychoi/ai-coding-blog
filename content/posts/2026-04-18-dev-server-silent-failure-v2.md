# 에러는 났고, 나는 몰랐고, 한 시간이 지났다

## 그냥 눈으로 확인하고 싶었을 뿐이었다

이 블로그는 내 컴퓨터에서 코드를 수정하고, 그걸 인터넷에 올리는 방식으로 운영된다. 그런데 수정한 게 실제로 어떻게 보이는지 확인하려면, 인터넷에 올리기 전에 **로컬 서버**를 켜서 먼저 확인할 수 있다.

> 로컬 서버란? 인터넷에 올리기 전에 내 컴퓨터 안에서만 돌아가는 임시 사이트다. 마치 공사 중인 건물을 외부에 공개하기 전에 내부에서만 둘러보는 것처럼. 주소가 `http://localhost:3011` 같은 형태인데, "localhost"는 "내 컴퓨터"라는 뜻이다.

`npm run dev`라는 명령어로 이 로컬 서버를 켤 수 있다. 평소엔 10초 안에 켜졌다. 그래서 아무 생각 없이 실행했다. 글 하나 수정했고, 어떻게 보이는지 확인하면 끝인 줄 알았다.

---

## ✓ Starting... 그리고 침묵

명령어를 실행하면 **터미널**에 이런 메시지가 뜬다.

> 터미널이란? 코드로 컴퓨터에 명령을 내리는 검은 창이다. 마우스로 클릭하는 대신 글자를 입력해서 컴퓨터를 조작한다.

```
▲ Next.js 16.1.6 (Turbopack)
- Local: http://localhost:3011

✓ Starting...
```

그리고 아무 일도 일어나지 않았다. `✓ Starting...` 은 계속 떠 있는데, 브라우저에서 주소를 열면 "사이트에 연결할 수 없음"만 나왔다. ~~체크 표시가 있으니까 성공한 거 아닌가?~~ 아니었다.

30초 기다렸다. 1분 기다렸다. 혹시 느린 건가 싶어서 5분을 기다렸다. 여전히 "연결할 수 없음". 이게 성공인지 실패인지조차 모르는 상태로 시간이 흘렀다.

## 내가 시도한 것들 (모두 실패)

### 포트 번호 바꾸기

**포트** 번호가 겹치는 거 아닌가 싶었다. 다른 프로그램이 3011번을 먼저 쓰고 있으면 막힐 수 있으니까.

> 포트란? 컴퓨터의 출입구 번호다. 같은 컴퓨터에서 여러 서버를 동시에 켤 때 번호로 구분한다. 3011, 3000, 8080 같은 숫자가 포트 번호다.

3011 → 3012 → 3013... 3019까지 전부 바꿔봤다. **전부 `✓ Starting...`에서 멈췄다.** 9개 포트를 순서대로 다 돌렸는데 결과가 동일했다. 뭔가 근본적으로 잘못됐다는 느낌이 처음 들었다.

### 캐시 삭제

`.next` 폴더에 이전 작업 결과물들이 **캐시**로 저장돼 있는데, 이게 오래되거나 깨지면 문제가 생기기도 한다.

> 캐시란? 한번 처리한 것들을 저장해두는 임시 저장소다. 다음에 같은 걸 빠르게 불러오려고 저장해두는 것인데, 이게 오래되거나 깨지면 오히려 문제를 일으키기도 한다.

```bash
rm -rf .next && npm run dev -- --port 3011
```

마찬가지였다. 캐시를 지우니까 처음부터 다시 만드느라 오히려 더 느려지기만 했다. 해결은커녕 기다리는 시간만 늘어났다.

### 빌드 도구 바꾸기

Next.js 16은 *Turbopack*이라는 새 **빌드 도구**를 기본으로 쓴다.

> 빌드 도구란? 내가 쓴 코드를 브라우저가 이해할 수 있는 형태로 변환해주는 도구다. 공장에서 원재료를 완성품으로 바꾸는 컨베이어 벨트 같은 것.

새 도구니까 불안정한 거 아닌가 싶어서 `--no-turbo`를 붙여봤다. `unknown option` 에러가 났다. `--webpack`으로 다시 시도했더니 이번엔 실행은 됐는데 **똑같이 멈췄다.** 빌드 도구를 바꿨는데도 결과가 같았다. *솔직히 이쯤에서 컴퓨터를 닫고 싶었다.*

---

## 진짜 원인은 숨겨진 에러였다

한 시간이 지나도록 아무것도 안 됐다. 그때 Claude가 다른 방법을 꺼냈다. 터미널 메시지에는 사실 두 종류가 있는데 — **stdout**과 **stderr** — 이걸 분리해서 따로 저장해보자는 거였다.

> stdout이란? 프로그램이 정상적으로 출력하는 메시지 채널이다. 화면에 보이는 대부분의 메시지가 여기서 나온다.

> stderr란? 에러 메시지만 따로 출력하는 채널이다. 보통 터미널에서는 stdout과 섞여 보이지만, 의도적으로 분리해서 볼 수 있다.

```bash
node node_modules/.bin/next dev --port 3019 1>/tmp/stdout.txt 2>/tmp/stderr.txt
```

stderr 전용 파일을 열었더니:

```
⨯ Unable to acquire lock at .next/dev/lock,
  is another instance of next dev running?
```

**한 줄이었다.** 그것도 딱 한 줄. 이 에러가 처음부터 계속 나고 있었는데, 터미널 화면에는 단 한 글자도 나타나지 않은 거였다. 에러가 없는 게 아니라 — 에러가 보이지 않는 곳에서 나고 있었다. ~~한 시간 동안 멀쩡한 척하고 있었던 거다.~~

---

## 자물쇠 파일이 남아 있었다

`next dev`는 실행될 때 `.next/dev/lock`이라는 **자물쇠 파일**을 하나 만든다. "나 지금 쓰고 있음" 표시다. 공용 화장실 문에 걸어두는 사람 있음 표시 같은 것.

서버를 정상적으로 끄면 이 파일도 같이 지워진다. 그런데 `pkill`로 **프로세스**를 강제로 죽이면 파일이 그대로 남는다.

> 프로세스란? 컴퓨터에서 돌아가고 있는 프로그램 하나하나를 말한다. `pkill`은 특정 프로세스를 강제로 종료하는 명령어다.

화장실 문 열쇠를 안에 둔 채로 창문으로 탈출한 셈이다. 그 다음 사람은 문이 잠겨 있으니 들어가지 못한다.

`next dev`를 다시 실행하면 이 자물쇠 파일을 발견하고 → 조용히 포기한다. "포기했다"는 말은 stderr에만 쓰기 때문에, 화면엔 아무것도 안 나온다. 이 세션에서 서버를 껐다 켰다 반복할 때마다 파일이 쌓였고, 매번 조용히 포기가 반복되고 있었던 거다.

> !! 에러가 없는 게 아니었다. 에러가 보이지 않는 곳에서 나고 있었다.

---

## 해결은 한 줄이었다

```bash
pkill -9 -f "next" && rm -rf .next/dev/lock && npm run dev -- --port 3011
```

실행 중인 서버 프로세스 전부 종료, 자물쇠 파일 삭제, 서버 재시작. 한 번에 됐다. 한 시간을 써서 얻은 해결책이 명령어 한 줄이라는 게 좀 허탈했지만, 그게 또 이런 거다.

한 가지 더. 이렇게 깨끗하게 시작해도 **첫 실행은 8분이 걸렸다.** `✓ Starting...` 이후로도 한참을 기다려야 진짜 켜진다. 두 번째부터는 1초도 안 걸린다. Next.js 16의 새 빌드 도구가 첫 번째 변환에 유독 오래 걸리는 것 같다. 이것도 몰라서 "또 멈췄나" 하고 일찍 포기했던 게 여러 번 있었다.

## 다음에 또 안 열리면

```bash
# 자물쇠 파일 확인
ls .next/dev/lock

# 있으면 지우고 재시작
pkill -9 -f "next" && rm -rf .next/dev/lock && npm run dev -- --port 3011

# 에러 메시지 직접 확인하고 싶으면
npm run dev 2>/tmp/err.txt & sleep 5 && cat /tmp/err.txt
```
