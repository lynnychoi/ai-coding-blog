# 오늘 내가 한 실수들 (그리고 어떻게 고쳤는지)

오늘 히어로 섹션 하나 만들면서 예상보다 훨씬 많은 걸 고쳤다. 잘 된 것보다 안 된 것들이 더 많이 쌓였는데, 그게 사실 더 남는 게 많다는 걸 알아서 기록해둔다.

이 글은 기획자/디자이너가 개발을 처음 배울 때 부딪힐 수 있는 것들이라 좀 자세히 써뒀다. 용어 설명도 최대한 쉽게 달아뒀으니 참고.

![여기서부터 시작](https://media.giphy.com/media/VXCPgZwEP7f1e/giphy.gif)

## 1. 포트 충돌: 모바일에서 아무것도 안 보였다

**상황:** 개발 중인 블로그를 모바일로 확인하고 싶어서 localtunnel을 썼다. 근데 모바일에서 아무것도 안 보였다. 화면이 그냥 하얀 배경만 나왔다.

> **localtunnel?** 내 컴퓨터에서 개발 중인 웹사이트를 외부(핸드폰, 다른 기기)에서 볼 수 있게 임시로 URL을 만들어주는 도구. 배포 전에 모바일 화면을 확인할 때 쓴다.

> **포트?** 컴퓨터에서 프로그램이 외부 연결을 기다리는 "번호판" 같은 것. 3000번 포트에서 실행 중이라는 건, 3000번 창구에서 손님을 받고 있다는 의미. 여러 프로그램이 각자 다른 번호를 쓴다.

**왜 그랬나:** Next.js를 켰을 때 3000번이 이미 사용 중이라서 자동으로 3002번으로 올라갔다. 근데 나는 localtunnel을 3000번으로 연결했다. 프로그램은 3002번 창구에 있는데 3000번 창구를 보고 있으니 당연히 아무것도 없는 것.

이유도 모르고 한참 다른 데서 원인을 찾았다. 터미널 로그를 좀 더 꼼꼼히 봤으면 바로 찾았을 것.

**고친 방법:** 실행 중인 프로세스 전부 종료하고 처음부터 다시 시작.

```bash
# 3000번 포트를 쓰는 프로세스 강제 종료
lsof -ti:3000 | xargs kill -9
```

**배운 것:** 모바일에서 안 보이면 개발 서버가 몇 번 포트에 올라가 있는지 먼저 확인해.

## 2. 오로라 블롭이 모바일에서 사라졌다

**상황:** 데스크탑에서는 예쁘게 보이는 오로라 블롭이 모바일에서 확인하니 그냥 까만 배경만 보였다.

**왜 그랬나:** 히어로 섹션이 "최대 너비 860px" 제한 컨테이너 안에 들어가 있었다. 블롭은 그 너머까지 퍼져있었는데, 컨테이너가 그 바깥 부분을 가위로 자른 것처럼 잘라버린 것.

> **컨테이너(container)?** 웹 페이지에서 내용물이 너무 넓게 퍼지지 않도록 가운데 영역에 모아주는 틀. 긴 텍스트가 화면 끝까지 늘어나면 읽기 힘들어서 대부분 최대 너비를 정해둔다.

모바일은 화면 자체가 좁으니까 컨테이너가 거의 전체를 차지하고, 블롭이 그 바깥으로 나가 있어서 전부 잘린 거였다.

**고친 방법:** CSS "full-bleed" 기법. 히어로 섹션만 컨테이너 밖으로 꺼내서 화면 전체 너비를 차지하게 했다.

![해결 후 — 모바일에서도 블롭이 꽉 차게](/images/blog/hero-mobile.png)

```css
.hero {
  width: 100vw;            /* 화면 전체 너비 */
  margin-left: calc(50% - 50vw);  /* 컨테이너 기준 왼쪽으로 당겨서 화면 끝에 맞추기 */
}
```

텍스트는 안에 별도 래퍼를 두고 다시 가운데 정렬:

```css
.hero-content-wrap {
  max-width: 860px;
  margin: 0 auto;
  padding: 0 1.5rem;
}
```

> **vw?** viewport width의 약자. 화면 너비를 100으로 나눈 단위. `100vw` = 현재 화면 너비 전체. 화면이 375px이면 100vw = 375px, 1440px 화면이면 100vw = 1440px.

**배운 것:** 화면 전체를 써야 하는 섹션인데 컨테이너 안에 있으면 이 패턴으로 해결. 디자이너가 말하는 "풀블리드 배경"을 구현하는 방법이기도 하다.

## 3. 그라데이션 흰색이 안 보였다

![흰색 어디감](https://media.giphy.com/media/3ohhwMRUdrABMPWqOs/giphy.gif)

**상황:** 타이틀 "기획자가"에 민트에서 흰색으로 바뀌는 그라데이션을 넣었는데, 흰색이 거의 안 보였다.

**왜 그랬나:** 그라데이션은 텍스트 너비 전체에 걸쳐 색이 서서히 바뀐다. "기획자가"는 짧은 단어인데, 그라데이션이 "시작점 민트 → 끝점 흰색"으로만 설정돼 있으면 흰색은 텍스트 맨 오른쪽 끄트머리에서야 등장한다.

짧은 텍스트니까 그 끄트머리 부분이 아주 좁아서 흰색이 거의 보이지 않았던 것.

**고친 방법:** 중간에 경유 색을 하나 더 추가해서 흰색이 40% 지점에서 이미 시작하게 했다.

![완성된 타이틀 그라데이션](/images/blog/hero-title-gradient.png)

```css
/* 전: 끝에서야 흰색 */
background: linear-gradient(90deg, 민트색 0%, 흰색 100%);

/* 후: 40% 지점부터 흰색으로 가는 중간 색 추가 */
background: linear-gradient(90deg, var(--accent) 0%, #b0f0e0 40%, #ffffff 85%);
```

**배운 것:** 짧은 텍스트에 그라데이션 쓸 때는 중간 포인트를 직접 지정해야 한다. 포토샵/피그마에서 그라데이션 먹이는 거랑 같은 원리인데, CSS는 퍼센트로 중간 포인트를 직접 써줘야 한다.

## 4. 페이지 이동할 때 스크롤이 이상한 곳에 있었다

**상황:** 블로그 포스트 페이지에서 홈으로 돌아오면 스크롤이 중간쯤에 걸려 있었다. 글 읽다가 뒤로가기 누르면 페이지 중간이 보이는 것.

**왜 그랬나:** Next.js의 App Router(페이지 이동을 관리하는 시스템)가 특정 상황에서 스크롤 위치를 자동으로 맨 위로 초기화해주지 않는 경우가 있다. 이전 페이지의 스크롤 위치를 기억하고 있다가 그 위치로 돌아가는 것.

> **Next.js App Router?** Next.js에서 URL에 따라 어떤 화면을 보여줄지 결정하는 시스템. "이 주소로 오면 이 화면, 저 주소로 가면 저 화면"을 관리한다.

**고친 방법:** 페이지가 바뀔 때마다 자동으로 스크롤을 맨 위로 올려주는 작은 컴포넌트를 하나 만들었다.

```tsx
export default function ScrollTop() {
  const pathname = usePathname(); // 현재 URL 경로
  useEffect(() => {
    window.scrollTo(0, 0); // 페이지 바뀌면 맨 위로
  }, [pathname]);
  return null; // 화면에 아무것도 표시하지 않음
}
```

이걸 모든 페이지에 공통으로 들어가는 레이아웃 파일에 넣어두니 해결됐다.

**배운 것:** 당연히 될 것 같은 것도 직접 처리해야 할 때가 있다. "페이지 바뀌면 당연히 맨 위겠지"는 Next.js에서 항상 보장되지 않는다.

## 5. 블롭이 왼쪽에서만 맴돌았다

**상황:** 블롭들이 이미 데스크탑 큰 화면에서 보면 화면 왼쪽 구석에서만 왔다갔다 하고 있었다. 오른쪽 절반은 블롭이 한 번도 안 갔다.

**왜 그랬나:** 블롭의 이동 거리를 픽셀로 고정해뒀었다. "200px 오른쪽으로 이동" 이런 식으로. 1400px 너비 화면에서 200px은 전체의 14%밖에 안 된다. 화면이 커질수록 블롭이 상대적으로 더 조금 움직이는 것.

**고친 방법:** 이동 거리를 화면 비율(vw/vh)로 바꿨다.

```css
@keyframes floatBlob1 {
  0%   { transform: translate(0, 0) scale(1); }
  33%  { transform: translate(52vw, 4vh) scale(0.75); }  /* 화면의 52% 오른쪽으로 */
  66%  { transform: translate(60vw, 20vh) scale(0.85); }
  100% { transform: translate(0, 0) scale(1); }
}
```

> **@keyframes?** CSS 애니메이션에서 "이 시점에 이 위치에 있어라"를 정의하는 것. 0%는 시작, 100%는 끝. 중간 값들을 정해주면 그 사이를 알아서 부드럽게 이동한다.

추가로 블롭마다 애니메이션 시작 시점을 다르게 줬다. 세 블롭이 같은 박자로 움직이면 코레오처럼 규칙적으로 보인다. 시작점을 어긋나게 하면 각자 알아서 떠다니는 것처럼 보인다.

**배운 것:** 화면 크기에 반응해야 하는 건 픽셀 말고 vw/vh. 디자인으로 치면 px 단위 고정값 말고 %로 작업하는 것과 같은 맥락.

## 6. 모바일 히어로가 주소창 때문에 잘렸다

**상황:** 모바일에서 히어로 섹션이 화면보다 조금 길어서 스크롤이 생겼다. 화면 꽉 차게 만들고 싶었는데.

**왜 그랬나:** `height: 100vh`를 줬는데, `vh`는 브라우저 주소창 높이를 포함한 전체 창 높이를 기준으로 한다. 모바일에서 주소창이 있으면 실제 내용 영역은 그보다 작은데, 거기에 100vh를 채우려고 하니 넘치는 것.

> **vh?** viewport height. 화면 높이를 100으로 나눈 단위. 근데 모바일 브라우저는 주소창이 있어서 실제 보이는 영역이 더 좁다.

**고친 방법:** `svh`(small viewport height) 사용. 주소창이 있는 상태의 실제 보이는 영역을 기준으로 한다.

```css
.hero {
  min-height: 100svh; /* 주소창 있어도 딱 맞게 */
}
```

**배운 것:** 모바일 풀스크린에는 `vh` 대신 `svh`. 특히 iOS Safari 대응에서 중요하다.
