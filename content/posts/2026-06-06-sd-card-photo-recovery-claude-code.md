## 프로젝트 한 줄 소개

코딩 프로젝트가 아니다. 오늘 찍은 아기 사진을 SD 카드에서 실수로 삭제했고,
유료 복구 프로그램 없이 Claude Code랑 터미널만으로 전부 살려낸 기록.
'Claude Code는 개발 도구'라는 생각을 깨는 에피소드.

## 이 작업을 하게 된 이유

소니 카메라 SD 카드(128GB)에서 오늘 찍은 사진들을 실수로 지웠다.
지운 게 꽤 많았고, 그날 찍은 아기 사진이라 다시 찍을 수도 없는 것들.
Disk Drill, Wondershare Recoverit 같은 복구 프로그램을 먼저 다운받아 봤는데
— 스캔은 무료, 실제 복구는 유료(Disk Drill 기준 약 $89)였다.
혹시나 해서 Claude Code에 'SD 카드에서 이미지들을 실수로 삭제했는데 복구하고 싶어'라고 물었다.

## 뭘 했나 (작업 목록)

- Claude가 제일 먼저 한 말: **'지금부터 SD 카드에 아무것도 쓰지 마세요.'**
  삭제된 파일은 '빈자리' 표시만 된 상태라 덮어쓰기 전엔 살릴 수 있다고.
- `diskutil list`로 SD 카드 찾기 (`/dev/disk10`, exFAT, 123.8GB).
  여기서 단서 발견: 카드 빈 공간이 927MB뿐 → 지워진 파일이 그 안에 다 있다는 뜻.
- 무료 도구 3개를 단계적으로 시도:
  1. **PhotoRec** (무료 카빙 도구) — 설치까지 했지만 파일명/날짜를 못 살리는 방식
  2. **TestDisk Undelete** — 삭제 파일 목록을 원본 이름+날짜로 보여줌. 터미널 TUI를 직접 조작해야 해서 중도 포기
  3. **The Sleuth Kit** (디지털 포렌식 도구) — 최종 선택. 스크립트로 전부 자동화 가능
- `fls` 명령으로 삭제된 파일 5,056개 목록을 원본 경로+날짜와 함께 추출
- 오늘 날짜(2026-06-06)로 필터 → **정확히 32장의 JPG**만 골라냄
- `icat` 명령으로 32장 전부 복구. 원본 파일명(DSC08241.JPG ~ DSC08275.JPG) 그대로
- 검증 3중: 파일 크기가 원본 메타데이터와 일치 / 32장 전부 유효한 JPEG / EXIF 촬영시각 보존 (오전 10:51 ~ 오후 13:45)
- Claude가 복구된 사진을 직접 열어서 확인 — 카시트에 앉은 아기 사진이 멀쩡하게 나왔다
- 마지막으로 친구한테 공유할 수 있는 쉬운 가이드 문서까지 정리 (`~/Desktop/SD카드-사진복구-가이드.md`)

## 막혔던 것들 / 삽질

- **sudo 비밀번호의 벽.** Claude는 관리자 비밀번호를 입력할 수 없어서 디스크를 직접 못 읽었다.
  Claude Code 안에서 `!` 프리픽스로 sudo 명령을 실행하는 것도 두 번이나 조용히 실패
  (비밀번호 프롬프트가 안 떠서 실행 자체가 안 된 듯).
  → 결국 **별도 터미널 앱에서 `sudo chmod o+r /dev/disk10*` 한 줄**만 내가 직접 실행.
  '읽기 권한만 잠깐 열어주면 나머지는 내가 다 할게' 방식으로 역할 분담이 정리됐다.
- **TestDisk TUI에서 길을 잃음.** Claude가 '①로그 화면에서 No Log, ②Intel 선택…' 하고
  단계별 표까지 만들어줬는데, 실제로는 안내에 없던 디스크 선택 화면이 떴다.
  화살표 키로 5,000개 파일 중 오늘 것만 고를 생각을 하니 아득해서
  '자동으로 오늘 날짜인 것들 너가 선택 도와주면 안될까..?'라고 물었고,
  여기서 도구가 Sleuth Kit으로 바뀌면서 모든 게 풀렸다.
- **Claude도 삽질했다.** 복구 스크립트 첫 실행이 32장 전부 실패.
  원인이 웃긴데 — zsh에서 `path`는 PATH 환경변수와 연결된 특수 변수라서,
  Claude가 반복문 변수 이름을 `path`로 짓는 바람에 셸의 PATH가 통째로 날아가
  `awk`, `ls` 같은 기본 명령어조차 'command not found'가 됐다.
  변수명을 `fp`로 바꾸니 32장 전부 성공. AI도 사람처럼 변수명으로 사고를 친다.
- 파일 번호 중간에 빠진 3장(DSC08248, 08249, 08262)은 카드 어디에도 없었다.
  촬영 직후 카메라에서 바로 지웠거나 카메라가 번호를 건너뛴 것으로 추정. 미해결로 남김.

## 결정의 이유 (왜 이렇게 만들었나)

- **유료 프로그램 대신 무료 CLI 도구:** Disk Drill $89 vs 무료. 같은 원리로 동작한다.
- **PhotoRec(전부 복구 후 필터) 대신 Sleuth Kit(선택 복구):**
  내가 '오늘 찍은 사진만 먼저 선택해서 복구하는 게 낫지 않을까?'라고 물은 게 분기점.
  PhotoRec은 파일명을 `f0012345.jpg` 같은 임의 이름으로 복구하지만,
  Sleuth Kit은 exFAT의 삭제된 디렉토리 항목을 읽어서 **원본 파일명과 날짜로 골라서** 복구할 수 있다.
- **복구 파일은 무조건 Mac 쪽에 저장:** SD 카드에 쓰면 다른 삭제 파일을 덮어쓸 수 있어서.
- **권한은 읽기만(o+r):** 쓰기 권한은 안 열었고, 카드를 뺐다 꽂으면 자동으로 원상복구되는 방식.

## 그때 어떤 기분이었나

지운 직후엔 유료 프로그램 결제 직전까지 갔다. 혹시나 하고 Claude Code에 물어본 거였는데,
'빈 공간이 927MB뿐이니 지워진 파일이 다 그 안에 있다'는 분석을 듣고 처음으로 살릴 수 있겠다 싶었다.
TestDisk 화면에서 안내에 없는 화면이 떴을 때가 제일 답답했고, '자동으로 해주면 안될까'라고
반쯤 포기하듯 물었는데 그게 오히려 정답이었다.
마지막에 Claude가 복구된 사진을 열어서 보여줬을 때 — 카시트에 앉은 아기 얼굴이
그대로 나왔을 때 — 그때 안도했다. 0원으로, 한 시간 안에, 32장 전부.

## 배운 것 / Takeaway

- **파일 삭제는 책을 버리는 게 아니라 도서관 목록 카드에 '폐기' 도장만 찍는 것.**
  덮어쓰기 전에 멈추면 살릴 수 있다. 그래서 첫 행동이 '카드 그만 쓰기'여야 한다.
- 유료 복구 프로그램이 하는 일을 무료 포렌식 도구(Sleuth Kit)가 그대로 한다.
  fls(목록 보기)와 icat(꺼내오기) 두 명령이 전부였다.
- AI와의 역할 분담: Claude는 비밀번호가 필요한 일을 못 한다.
  '권한 한 줄만 사람이 열어주고 나머지는 AI가' 패턴은 다른 작업에도 쓸 수 있겠다.
- 막히면 도구를 바꿔달라고 말해도 된다. '이거 너가 자동으로 해주면 안 돼?'라는
  비개발자스러운 질문이 더 좋은 기술적 선택(TUI → 스크립트)을 끌어냈다.
- AI도 변수명 하나로 삽질한다. 에러가 나도 원인을 설명받으면 무섭지 않다.
- **핵심 한 줄:** 복구 프로그램에 $89 내기 전에, Claude Code에 한 번 물어볼 것 — 지운 사진은 아직 거기 있다."

## 개발 모르는 친구한테 먼저 설명하자면

파일을 삭제하면 실제로 데이터가 사라지는 게 아니다. 도서관에서 책을 버리는 게 아니라, 목록 카드에 "없음" 도장만 찍는 것과 같다. **책은 아직 그 자리에 있다.** 다만 그 자리에 다른 책이 꽂히는 순간 진짜로 사라진다.

그래서 사진을 지웠다면 그 SD 카드로 뭔가를 더 찍거나 저장하기 전에 멈춰야 한다. 자리를 덮어쓰기 전까지는 살릴 수 있다.

이걸 하는 프로그램을 보통 "복구 프로그램"이라고 부르는데, Disk Drill 같은 유명한 것들은 스캔은 무료지만 실제로 복구하려면 돈을 받는다. 오늘 내가 발견한 건 — 같은 원리의 무료 도구가 있고, Claude Code가 그걸 쓰는 방법을 처음부터 끝까지 다 알고 있다는 것.

## 어떻게 시작됐나

소니 카메라로 오늘 하루 찍은 사진들을 SD 카드에서 정리하다가 실수로 지웠다. 결혼식 사진이라 다시 찍을 수도 없는 것들.

처음엔 당연히 Disk Drill을 깔았다. 스캔을 돌렸더니 삭제된 파일들이 쭉 떴다. 삭제된 파일들을 미리보기까지 가능하니 일단 '어딘가에 남아있긴 하구나!'하는 마음에 안심이 됐다. 그리고 복구 버튼을 누르려는 순간 — *$89.* 결제하란다. 혹시나 하는 마음에 Claude Code를 켰다. "SD 카드에서 사진을 실수로 삭제했는데 복구하고 싶어."라고 입력했고, 그게 시작이었다.

## Claude가 제일 먼저 한 말

**"지금부터 SD 카드에 아무것도 쓰지 마세요."**

사진을 더 찍거나, 카드를 포맷하거나, 파일을 복사하거나 — 아무것도 하지 말라고. 이유는 위에서 설명한 것과 같다. 빈자리에 덮어쓰기 전까지가 승부다.

그다음으로 한 게 SD 카드 상태 파악이었다.

```
diskutil list
```

> 터미널이란? 코드로 컴퓨터에 명령을 내리는 검은 창이다. 마우스 대신 타이핑으로 컴퓨터와 대화하는 방식.

이 명령으로 SD 카드가 `/dev/disk10`이라는 주소에 128GB짜리로 연결돼 있다는 걸 확인했다. 그리고 결정적인 단서 하나 — **카드의 빈 공간이 927MB뿐이었다.** 128GB 카드가 거의 꽉 차 있다는 뜻이고, 그건 내가 지운 파일들이 아직 거기 다 있다는 증거였다. 덮어쓰인 게 없다.

그때 처음으로 *살릴 수 있겠다* 싶었다.

## 도구 세 개를 거쳤다

Claude는 단번에 정답 도구를 제시하지 않았다. 단계적으로 시도했고, 나는 중간에 한 번 막혀서 방향을 틀었다.

### 1차 시도: PhotoRec

Photorec은 무료 복구 도구다. 설치까지 했는데, 방식이 마음에 안 들었다. 이 도구는 삭제된 파일을 전부 꺼낸 뒤 `f0012345.jpg` 같은 임의의 이름을 붙인다. 원본 파일명도, 날짜도 사라진다. 오늘 찍은 것만 골라낼 방법이 없다.

"오늘 날짜인 것만 먼저 복구할 수 없을까?"라고 물었고 Claude가 다음 도구로 넘어갔다.

### 2차 시도: TestDisk

TestDisk는 삭제된 파일을 원본 이름과 날짜로 보여준다. 더 낫다. Claude가 단계별 표까지 만들어줬다.

그런데 실제로 열어보니 안내에 없던 화면이 먼저 떴다. 디스크 선택 화면이었는데, 거기서부터 이미 길을 잃었다. 5,000개 넘는 파일 목록에서 화살표 키로 오늘 것만 고른다는 게 아득하게 느껴졌다.

"자동으로 오늘 날짜인 것들 너가 선택 도와주면 안 될까..?"라고 반쯤 포기하듯 물었다.

그게 오히려 분기점이었다.



### 3차 시도: The Sleuth Kit — 이걸로 해결했다

> The Sleuth Kit이란? 원래 수사나 포렌식 목적으로 만들어진 무료 도구 모음이다. 삭제된 파일의 흔적을 읽는 데 쓴다. Disk Drill 같은 유료 복구 프로그램이 하는 일을 이 도구가 그대로 한다.

Claude가 "그러면 스크립트로 전부 자동화할 수 있는 도구로 바꾸자"고 했고, 설치 명령을 줬다.

```
brew install sleuthkit
```

여기서부터는 내가 명령어를 이해할 필요도 없었다. Claude가 스크립트를 짜줬고, 나는 붙여넣기만 했다.

## 실제 복구 과정 (재현 가능하게)

이 과정을 그대로 따라 하면 같은 상황에서 쓸 수 있다.

### Step 1 — SD 카드에 읽기 권한 열기

Claude는 관리자 비밀번호(sudo)가 필요한 명령을 직접 실행할 수 없다.

> sudo란? 컴퓨터의 관리자 권한으로 명령을 실행하는 것. 비밀번호를 입력해야 한다.

그래서 역할을 나눴다. 내가 터미널 앱을 별도로 열어서 딱 한 줄만 직접 입력했다:

```bash
sudo chmod o+r /dev/disk10s1
```

"읽기 권한만 잠깐 열어줘. 나머지는 내가 다 할게." 방식이다. 쓰기 권한은 건드리지 않았고, SD 카드를 뺐다 꽂으면 자동으로 원상복구된다.

### Step 2 — 삭제된 파일 목록 전체 추출

```bash
fls -r -m / /dev/disk10s1 > ~/Desktop/sd_files.txt
```

> fls란? The Sleuth Kit의 명령어 중 하나. 디스크에서 삭제된 파일 포함 전체 파일 목록을 뽑아준다.

이 명령 하나로 SD 카드의 삭제된 파일 **5,056개** 목록이 텍스트 파일로 나왔다. 원본 파일명, 경로, 날짜 전부 포함해서.

### Step 3 — 오늘 날짜 파일만 필터링

```bash
grep "2026-06-06" ~/Desktop/sd_files.txt | grep -i "\.jpg" > ~/Desktop/today_photos.txt
```

5,056개 중에서 오늘 날짜(2026-06-06)이고 JPG 파일인 것만 골랐다. **32개**가 나왔다. DSC08241.JPG부터 DSC08275.JPG까지.

### Step 4 — 32장 복구

Claude가 복구 스크립트를 짰다. 각 파일의 *inode 번호*를 읽어서 `icat` 명령으로 꺼내오는 방식이다.

> inode란? 파일 하나하나에 붙은 고유 번호. 파일명이 사라져도 이 번호로 실제 데이터를 찾을 수 있다.

> icat이란? inode 번호를 가지고 실제 파일 내용을 꺼내오는 명령어.

```bash
mkdir -p ~/Desktop/recovered_photos

while IFS='|' read -r meta path; do
    inode=$(echo "$meta" | awk -F'/' '{print $NF}' | tr -d ' ')
    filename=$(basename "$path")
    icat /dev/disk10s1 "$inode" > ~/Desktop/recovered_photos/"$filename"
done < ~/Desktop/today_photos_inodes.txt
```

첫 실행에서 32장이 전부 실패했다. 원인이 웃겼다 — Claude가 반복문 변수 이름을 `path`로 지었는데, *zsh에서 `path`는 시스템 PATH 환경변수와 연결된 특수 변수*라서 그걸 덮어쓰는 순간 `awk`, `ls` 같은 기본 명령어조차 "command not found"가 됐다. ~~AI도 사람처럼 변수명으로 사고를 친다.~~ 변수명을 `fp`로 바꾸니 32장 전부 성공.

### Step 5 — 검증

Claude가 세 가지를 확인했다:

- [x] 파일 크기가 원본 메타데이터와 일치
- [x] 32장 전부 유효한 JPEG 형식
- [x] EXIF 촬영 시각 보존 (오전 10:51 ~ 오후 13:45)

그리고 직접 이미지를 열어서 확인했다. 결혼식에서 찍었던 사진들이 우르르 나왔다.
