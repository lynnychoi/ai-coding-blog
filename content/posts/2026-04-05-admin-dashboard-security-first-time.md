  # 어드민 대시보드 + 보안

  ## 프로젝트 한 줄 소개
  Claude Code로 만드는 개인 블로그. 기획자 출신 Lynn이 AI 코딩을 처음 배우면서 겪는 것들을 기록. 글 관리는 전부 GitHub 마크다운 파일로.

  ## 이 작업을 하게 된 이유
  글을 쓰고 수정하려면 어디서든 접근할 수 있는 관리 페이지가 필요했다. 근데 이게 공개되면 안 된다. 내 블로그인데 남이 글을 올릴 수 있으면 곤란하니까. 동시에 글 목록도 한눈에 보고 싶었다. "이 글 수정하고 싶다" 싶을 때 바로 접근할 수 있게.

  ## 뭘 했나
  - 어드민 대시보드 (`/admin`) — dev/writing 타입별로 글 목록 보여주고, 각 글마다 "보기" + "수정" 링크
  - `app/admin/LogoutButton.tsx` — 서버 컴포넌트인 대시보드에서 클라이언트 동작(로그아웃)이 필요해서 컴포넌트 분리
  - `app/api/posts/route.ts` — 글 목록 JSON API
  - `app/api/admin/giphy/route.ts` — Giphy API를 서버사이드에서 프록시. API 키를 클라이언트에 노출하지 않으려고.
  - `middleware.ts`로 `/admin` 전체 경로 보호 — 쿠키 없으면 로그인 페이지로

  ## 막혔던 것들 / 삽질
  - **middleware.ts 이름 deprecated**: Next.js 16에서는 `proxy.ts`로 이름을 바꿔야 한다고 경고가 뜬다. 빌드는 됐는데 찝찝함. 아직 안 고쳤다.
  - **서버/클라이언트 컴포넌트 분리**: 어드민 대시보드를 서버 컴포넌트로 만들었는데, 로그아웃 버튼이 `onClick`이 필요해서 클라이언트 컴포넌트여야 했다. 처음엔 전체를 클라이언트로 만들려다가, 그러면 서버사이드 데이터 패칭을 못 쓰니까 버튼만 분리했다.
  - **dev server lock 파일 충돌**: 작업하다가 `next dev`가 이미 실행 중인데 또 실행했더니 `.next/dev/lock` 충돌. `pkill -f "next dev"` 로 해결.

  ## 결정의 이유
  - Giphy API를 서버사이드 프록시로 만든 건 API 키 노출 방지. 클라이언트에서 직접 호출하면 브라우저 개발자도구에서 키가 보인다.
  - 대시보드를 서버 컴포넌트로 유지한 건 글 목록을 서버에서 바로 읽어오기 위해. 클라이언트에서 fetch하면 깜빡임이 생긴다.

  ## 그때 어떤 기분이었나
  보안 설계가 생각보다 고민할 게 많았다. Edge 런타임이니 Node 런타임이니 처음 보는 개념들이 나오는데, 왜 이걸 구분해야 하는지 감이 안 잡혔다. Claude가 설명해줬는데 이해는 했지만 직접 부딪히지 않으면 몰랐을 것들. middleware 경고는 아직 신경 쓰인다.

  ## 배운 것 / Takeaway
  - Next.js에서 서버 컴포넌트와 클라이언트 컴포넌트를 섞을 때는, 클라이언트 동작이 필요한 부분만 분리하는 게 맞다
  - API 키는 항상 서버사이드에서만 쓸 것. 프록시 라우트 만드는 게 귀찮아도 해야 한다
  - **핵심 한 줄**: 기획자가 "보안"을 구현한다는 게 어색했는데, 개념을 이해하고 나면 코드는 Claude가 써줘도 된다.

# 보안이 뭔지도 모르면서 어드민 페이지 만든 날

글이 쌓이기 시작하면서 문제가 생겼다. 수정하고 싶은 글이 생겼을 때 매번 GitHub에 들어가서 파일을 찾는 게 번거로웠다. 관리 페이지가 필요했다. 어디서든 접근할 수 있고, 글 목록을 한눈에 볼 수 있고, 수정 링크를 바로 누를 수 있는 것.

근데 그게 공개되면 안 된다. 내 블로그인데 남이 글을 올릴 수 있으면 곤란하니까.

"보호해야 한다"는 건 알겠는데, 어떻게 보호하는 건지는 몰랐다.

![어드민 만들기 전 나의 마음](https://media.giphy.com/media/Fn9PEDVnNT3Ik/giphy.gif)

## 일단 뭘 만들었냐면

크게 세 가지다.

**어드민 대시보드** (`/admin`). dev 타입 글, writing 타입 글을 구분해서 목록으로 보여주고, 각 글마다 "보기"와 "수정" 링크가 붙어 있다. 이게 있으면 GitHub 안 들어가도 된다.

**글 목록 API** (`/api/posts`). 대시보드가 글 목록을 불러오는 창구다.

> **API란?** 두 프로그램이 서로 대화하는 창구다. 여기선 대시보드 화면이 서버한테 "글 목록 줘"라고 요청하면, 서버가 JSON 형태로 돌려주는 것.

**미들웨어로 경로 보호** (`middleware.ts`). `/admin`으로 시작하는 모든 주소에 접근할 때 로그인 쿠키가 있는지 확인한다. 없으면 로그인 페이지로 보낸다.

> **미들웨어란?** 요청이 페이지에 도달하기 전에 중간에서 가로채는 것. 경비원 같은 역할이다. "신분증 있어요?" 하고 물어보고, 없으면 못 들어오게 막는다.

## 삽질 기록

### 로그아웃 버튼 하나 때문에

대시보드를 *서버 컴포넌트*로 만들었다.

> **서버 컴포넌트란?** Next.js에서 페이지를 서버에서 미리 완성해서 브라우저로 보내는 방식. 데이터를 서버에서 바로 읽어올 수 있어서 화면 깜빡임이 없다.

그런데 로그아웃 버튼이 문제였다. 버튼을 누르면 뭔가 일어나야 하는데, 이런 *클릭 이벤트*는 클라이언트 컴포넌트에서만 쓸 수 있다. 서버 컴포넌트에는 `onClick`이 없다.

~~처음엔 대시보드 전체를 클라이언트 컴포넌트로 바꾸면 되겠다 싶었다.~~ 근데 그러면 서버에서 데이터 읽어오는 장점을 다 버리는 거라 이상했다. Claude한테 물어봤더니 버튼만 분리하면 된다고 했다. `LogoutButton.tsx` 파일 하나 만들어서 거기서만 클라이언트 동작을 처리하고, 대시보드 본체는 서버 컴포넌트 유지.

생각해보면 당연한 구조인데, 직접 부딪혀보지 않으면 감이 안 잡히는 것들이 있다.

### middleware.ts 경고

빌드하니까 경고가 떴다. Next.js 16부터는 `middleware.ts` 이름이 deprecated됐고 `proxy.ts`로 바꿔야 한다는 내용이었다.

빌드는 됐다. 작동도 한다. 근데 찝찝하다. 아직 안 고쳤다. 이 글 쓰는 지금도 신경 쓰인다.

### dev server lock 충돌

작업하다가 터미널 창 두 개에서 `next dev`를 동시에 실행했더니 `.next/dev/lock` 충돌이 났다. 해결은 간단했다.

```bash
pkill -f "next dev"
```

프로세스 다 죽이고 다시 실행. 끝.

## API 키는 왜 숨겨야 하나

Giphy API를 쓰는데, API 키를 클라이언트에서 직접 호출하면 브라우저 개발자도구를 열면 키가 그대로 보인다.

> **API 키란?** 서비스를 쓸 때 발급받는 비밀번호 같은 것. 이게 노출되면 남이 내 계정으로 API를 마구 써버릴 수 있다.

그래서 `/api/admin/giphy` 라는 *프록시* 라우트를 만들었다.

> **프록시란?** 중간에서 대신 요청해주는 것. 브라우저 → 내 서버 → Giphy 순서로 거치면, 브라우저는 Giphy API 키를 볼 수 없다.

귀찮긴 했다. 파일 하나 더 만들어야 하니까. 근데 해야 하는 거라는 건 이번에 제대로 이해했다.

## 기획자가 보안을 구현한다는 것

이번 작업에서 제일 낯선 부분이 보안이었다. Edge 런타임이니 Node 런타임이니 처음 보는 개념들이 쏟아졌다. 왜 구분해야 하는지, 어디서 어떤 걸 쓸 수 있는지 감이 없었다.

Claude가 설명해줬고, 이해는 했다. 코드는 Claude가 썼다. 근데 "왜 이렇게 해야 하는지"를 이해하지 못하면 어디서 뭘 물어봐야 하는지도 모른다. 그 이해 부분은 내가 해야 했다.

> 개념을 이해하고 나면 코드는 Claude가 써줘도 된다. 순서가 중요하다.
